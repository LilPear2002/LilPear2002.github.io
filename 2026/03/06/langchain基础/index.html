LangChain基础

LangChain自身并不开发LLMs，它的核心理念是为各种LLMs实现通用的接口，把LLMs相关的组件“链接”在一起，简化LLMs应用的开发难度,方便开发者快速地开发复杂的LLMs应用.LangChain是一个开发LLM相关业务功能的集大成者，是一个Python的第三方库，提供了各种功能的API.包括：

•提示词优化的相关功能API

•调用各类模型的功能API

•会话记忆的相关功能API

•各类文档管理分析的功能API

•构建Agent智能体的相关功能API

•各类功能链式执行的能力

Models

LangChain模型组件提供了与各种模型的集成，并为所有模型提供一个精简的统一接口。LangChain目前支持三种类型的模型：LLMs（大语言模型）、Chat Models(聊天模型)、Embeddings Models(嵌入模型）.

•LLMs:是技术范畴的统称，指基于大参数量、海量文本训练的 Transformer 架构模型，核心能力是理解和生成自然语言，主要服务于文本生成场景

•聊天模型:是应用范畴的细分，是专为对话场景优化的 LLMs，核心能力是模拟人类对话的轮次交互，主要服务于聊天场景

•文本嵌入模型: 文本嵌入模型接收文本作为输入, 得到文本的向量.

LangChain支持的三类模型，它们的使用场景不同，输入和输出不同，开发者需要根据项目需要选择相应。我们所用的阿里云通义千问系列主要来自于：langchain_community包

llms

•invoke，调用模型，一次型返回完整结果

•stream，调用模型，逐段流式输出

这两个方法是新版LangChain(1.0版本后)中基于Runnable接口的通用核心方法。绝大多数组件（如提示词模板、链、向量检索、工具调用等，后续学习）都支持这两个方法，这也是 LangChain 设计的核心统一范式。

chat_models

聊天消息包含下面几种类型，使用时需要按照约定传入合适的值：

•AIMessage: 就是 AI 输出的消息，可以是针对问题的回答. (OpenAI库中的assistant角色）

•HumanMessage: 人类消息就是用户信息，由人给出的信息发送给LLMs的提示信息，比如“实现一个快速排序方法”. (OpenAI库中的user角色）

•SystemMessage: 可以用于指定模型具体所处的环境和背景，如角色扮演等。你可以在这里给出具体的指示，比如“作为一个代码专家”，或者“返回json格式”. (OpenAI库中的system角色）

Chains

「将组件串联，上一个组件的输出作为下一个组件的输入」是 LangChain 链（尤其是 | 管道链）的核心工作原理，这也是链式调用的核心价值：实现数据的自动化流转与组件的协同工作如下

chain = prompt_template | model

核心前提：即Runnable子类对象才能入链（以及Callable、Mapping接口子类对象也可加入（后续了解,用的不多））。目前所学习到的组件，均是Runnable接口的子类，如下类的继承关系：

•是Runnable接口的直接子类

•也是绝大多数组件的父类

•通过invoke或stream进行阻塞执行或流式执行

|运算符重写

前文代码中： chain = chat_prompt_template | model

在语法上使用了|运算符的重写

在 Python 中，运算符（如 +、|）的行为由类的魔法方法决定。例如：

•a + b 本质调用的是 a.add(b)

•a | b 本质调用的是 a.or(b)

只需要自行实现类的__or__方法，即可对|符号的功能进行重写。

示例： 让 a|b|c 的代码得到一个自定义的类对象(类似列表即[a, b, c]) ,调用run方法依次输出a、b、c,需要重写 | 即 __or__方法

knowledge_base.py:

  • MD5 去重机制 (check_md5, save_md5, get_string_md5)：
    • 原理：对每一份上传的文本内容计算 MD5 哈希值。
    • 作用：在数据库中建立“指纹库”。如果用户上传了完全相同的内容，系统会直接跳过，避免向量库中产生冗余数据，节省 Token 和存储空间。
  • KnowledgeBaseService 类：
    • 初始化 (init)：配置 Chroma 向量库（负责存取向量）和 RecursiveCharacterTextSplitter（负责把长文本切成小块）。
    • 切分逻辑 (upload_by_str)：
      • 分段处理：如果文本超过阈值，使用 RecursiveCharacterTextSplitter 进行切分。切分时保留 chunk_overlap（重叠部分），这是 RAG 中的关键策略，为了防止语义在切分处被割裂。
    • 元数据注入 (metadata)：除了文本本身，还给每个切片打上了 source（文件名）、create_time（时间戳）和 operator（操作员）标签。这在后续检索时非常有用，可以方便地告诉用户“这段话来自哪个文件”。
    • 存入数据库 (chroma.add_texts)：将切分后的文本转化为向量并保存到 persist_directory 目录。

app_file_uploader.py:

  • 响应式 UI：Streamlit 的特性是“页面变动则代码重跑”。
  • st.session_state：这是 Web 应用的持久化关键。service 对象被存入 session_state 后，用户在页面上进行多次操作时，KnowledgeBaseService 实例不会被重复创建，从而保证了数据库连接的稳定性。
  • 交互逻辑：
    1. 用户上传文件。
    2. 读取文件内容并解码为 utf-8 字符串。
    3. 通过 st.spinner 给用户提供反馈（转圈动画），避免用户在处理大文件时以为页面卡死了。
    4. 调用后台的 upload_by_str 方法，并将结果展示在网页上。

file_history_store.py：

  • FileChatMessageHistory: 它继承自 LangChain 的 BaseChatMessageHistory。
  • 核心逻辑: 它将对话记录以 JSON 格式存储在本地文件（./chat_history/session_id）中。
  • 关键方法:
    • add_messages: 将新的对话消息追加到旧记录中，并重新写入磁盘。
    • messages 属性: 自动从磁盘读取 JSON，并利用 messages_from_dict 将其还原为 LangChain 可识别的 BaseMessage 对象。
  • 意义: 即使重启程序，之前的对话记录依然存在，实现了“会话保持”。

vector_stores.py:

它封装了 Chroma 向量库。get_retriever(): 返回一个检索器，它会自动处理用户输入，将其向量化，并在向量数据库中匹配最相关的 k 个文档片段。

rag.py:

  • RagService: 负责将所有组件串联。
  • __get_chain (核心中的核心):
    • 这里使用了 LCEL 链式语法。
    • 数据流转:
      1. 并行处理: RunnablePassthrough 和 retriever 并行工作，分别获取用户的原始输入和检索到的文档片段（context）。
      2. 格式化: format_document 将文档对象转为人类可读的字符串。
      3. Prompt 组装: format_for_prompt_template 将输入、检索到的上下文和历史记录整理成 Prompt 所需的字典格式。
    • RunnableWithMessageHistory: 这是 LangChain 处理多轮对话的神器。它自动调用 get_history 来加载该 session_id 的历史记录，并将它们注入到 MessagesPlaceholder(“history”) 中。

app_qa.py:

  • Streamlit 的使用:
    • st.chat_message 和 st.chat_input：创建了一个类似 ChatGPT 的聊天界面。
    • st.session_state: 用来缓存 RagService 实例和对话消息记录，确保页面重绘时应用不崩溃。
  • 流式输出 (write_stream):
    • 这是用户体验的关键。它并没有等待整个 AI 回答完毕才显示，而是通过 res_stream 实时将 AI 的回复流式打印在页面上。
    • capture 函数是一个巧妙的“拦截器”，它在流式输出的同时，将每一块内容存入 ai_res_list，最后拼凑在一起存入历史记录。

Middleware中间件

  • 生命周期钩子 (before_agent, after_agent, before_model, after_model)：
    这些函数在 Agent 或 Model 执行的特定时间点触发。它们就像是“打卡机”，告诉你程序执行到哪一步了。
    • state 参数：非常关键，它包含了当前所有的对话历史，你可以通过它实时获取模型的状态。
    • runtime 参数：提供了当前执行环境的上下文信息。
  • 包装器（wrap_model_call, wrap_tool_call）：
    这是更底层的控制。通过 handler，可以完全控制请求的发送过程。
    • handler(request)：这是真正的执行动作。
    • 你可以在 handler 执行之前打印日志（监控），甚至在执行之后修改模型返回的数据（比如过滤掉敏感词）。
