requests库基础学习

基础

接口测试本质上就是客户端（你的代码） 发送请求，服务器 处理并返回响应。

GET 请求

GET 请求通常用于“获取”数据（比如查询商品列表）。在接口自动化中，GET 请求最常见的场景是：带参数查询。

httpbin.org 是一个专门用来练习 HTTP 请求的网站，它会把发送给它的东西原样返回，适合用来调试代码。

POST 请求

POST 请求通常用于“提交”数据（比如登录、注册、创建订单）。这是测试中最容易出错的地方，因为数据格式必须准确。

区分两种最常见的数据格式：

  1. Form 表单 (application/x-www-form-urlencoded)：传统的网页提交方式。
  2. JSON (application/json)：现代前后端分离接口（RESTful API）最常用的方式。****

解析响应

属性 说明 测开用途
response.status_code HTTP 状态码 第一层断言：先判断是不是 200。
response.json() 解析 JSON 响应 核心断言：获取业务字段（如 code, msg, token）。
response.text 原始文本 当返回不是 JSON（如 HTML 或报错信息）时查看。
response.elapsed 响应耗时 用于简单的性能监控（如断言响应必须 < 500ms）。

身份伪装与会话管理

在真实的测试场景中，大部分接口不是“裸奔”的。比如：

  1. 有些接口必须登录后才能访问（需要 Cookie 或 Token）。
  2. 有些接口限制了客户端类型（需要 User-Agent）。
  3. 接口测试通常是连续的：先登录 -> 再查询 -> 再下单。

“如何让代码像浏览器一样，记住登录状态？”

定制请求头 (Headers)

很多时候直接请求接口会返回 403 Forbidden，或者返回的数据不对，大概率是因为缺少了 Headers。

在自动化框架中，有两个 Header 最常用：

  • User-Agent: 告诉服务器你是谁（浏览器？手机？还是爬虫脚本）。
  • Authorization / token: 用于鉴权（告诉服务器你登录了）。

测试开发视角：在封装框架时，会把 headers 放在公共配置里（比如 config.py），这样不用每个测试用例都手动写一遍 User-Agent。

会话对象 (Session)

问题场景：假设有两个接口：

  1. /login (登录成功后，服务器会给浏览器发一个 Cookie)
  2. /profile (查询个人信息，需要带着刚才那个 Cookie 才能查)

如果直接用 requests.post 登录，再用 requests.get 查信息，第二次请求会失败。因为 requests 是“无状态”的，第二次请求时它已经忘记了第一次请求发生的事。

解决方案：requests.Session()。Session 对象就像一个浏览器，它会自动帮你保存和携带 Cookie。

在未来写的自动化框架中，通常会创建一个 BaseApi 类，并在 init 方法里实例化 self.session = requests.Session()。所有的接口请求都通过 self.session 发送，从而保证全流程登录状态不丢失。

超时处理 (Timeout)

自动化测试要想稳定，必须要有超时控制。如果服务器挂了，脚本不能一直卡在那里等死。
