Fiddler基础学习

Fiddler 是一款免费的 HTTP/HTTPS 抓包与调试代理工具，主要用于Web和移动端应用的网络请求分析。它运行在Windows系统上，通过设置自身为系统代理来捕获所有经过的HTTP/HTTPS流量

界面熟悉

1. 界面分区

  • 左侧（Session List）：显示所有抓到的请求会话。关注 Result（状态码）、Host（主机名）、URL。
  • 右侧（Inspectors）：查看请求和响应的详细内容。
    • Request（上半部分）：查看请求头、Cookies、请求体（Raw/JSON）。
    • Response（下半部分）：查看响应头、响应体（Raw/JSON/XML）。

2. 过滤干扰（Filters） 如果不设置过滤，系统 updates、聊天软件的包都会混在一起。

  • 操作路径：右侧选项卡点击 Filters -> 勾选 Use Filters。
  • 设置：在 Hosts 区域，选择 Show only the following Hosts。
  • 输入：输入你想抓的域名（例如 www.baidu.com 或你的测试环境 IP）。
  • 生效：点击 Actions -> Run Filterset now。

基础操作

过滤请求

如果不设置过滤，Fiddler 会抓到电脑上所有软件（微信、Windows更新、杀毒软件）的流量，非常乱。通过右侧的 Filters（过滤器） 选项卡来操作：

第一步：开启过滤

  1. 点击右侧选项卡中的 Filters。
  2. 勾选最顶部的 Use Filters（勾选后，选项卡标题可能会变色，提醒你过滤已开启）。

第二步：设置过滤规则（Hosts）

找到 Hosts 区域，这里决定了我们要看谁的请求。

  • 场景 A：只看特定网站（最常用）

    1. 在下拉框选择 Show only the following Hosts（只显示以下主机）。
    2. 在下方输入框填写域名。支持多个域名（用分号隔开）和通配符。
    • 示例：你想抓百度的包，输入：www.baidu.com; *.baidu.com
    • 解释：*.baidu.com 表示匹配 map.baidu.com, https://www.google.com/search?q=tieba.baidu.com 等所有二级域名。

  • 场景 B：排除特定网站

    1. 在下拉框选择 Hide the following Hosts。
    2. 输入你想屏蔽的域名（比如屏蔽掉无关的广告域名）。

第三步：使规则生效

  • 点击 Filters 面板右上角的 Actions 按钮。
  • 选择 Run Filterset now。
  • 左侧列表中，不符合规则的请求瞬间消失了（变灰或被隐藏）

控制抓取

在测试过程中，通常只在需要的时候开启抓包。比如只想测“点击登录”这一个动作，那就先暂停抓包，清空面板，准备好后再开启。

  • 状态查看：看 Fiddler 界面左下角。
    • 显示 Capturing（有图标）：表示正在抓包。
    • 空白（无图标）：表示停止抓包。
  • 快捷操作：
    • 鼠标点击：直接点击左下角的 Capturing 区域来切换开关。
    • 快捷键：按下键盘上的 F12。这是最常用的快捷键，按一下开，再按一下关。
    • 菜单栏：File -> Capture Traffic（打钩表示开启）。

测试场景：当准备复现一个 Bug 时，先按 F12 停止抓包，清理掉无关数据，再按 F12 开启，操作 APP/网页，操作完立刻 F12 停止。这样抓到的包最纯净。

删除请求

为了防止看花眼，需要频繁清理 Session 列表（左侧的会话列表）。

  • 常用操作：
    • 清空所有：点击工具栏上的 X 图标（Go 按钮旁边），或者使用快捷键 Ctrl + X。
    • 删除选定：选中一条或多条请求（按住 Shift 或 Ctrl 多选），按键盘 Delete 键。
  • 高级删除（右键菜单）：
    • 在列表右键 -> Remove -> Images
    • 在列表右键 -> Remove -> Unselected Sessions（只保留选中的，其他的全删掉）。
  • 命令行写法：
    • 在左下角黑色小框（QuickExec）输入 cls 然后回车，也可以清空列表（Clear Screen 的缩写）。

基础定位与解析

打开百度搜索HelloWorld：

百度是一个极其复杂的网站，当点击“百度一下”时，它会在后台发送几十个请求（联想词、图片、统计埋点、广告等）。

如何快速找到你要的那个“搜索请求”？即利用关键字搜索和查看参数。

操作演练：

  1. 清空面板：按 Ctrl + X，让世界清静一下。
  2. 触发请求：在百度搜索框输入HelloWorld，然后点击“百度一下”。
  3. 停止抓包：页面加载完立刻按 F12 停止。
  4. 查找（Find）：
    • 在 Fiddler 界面按 Ctrl + F。
    • 在弹出的框里输入你刚才搜的词：test12345。
    • 点击 Find Sessions。
  5. 观察结果：
    • Fiddler 会把包含这个词的请求高亮显示（通常是黄色）。
    • 找到高亮的那个请求，看它的 URL。
    • 百度的搜索接口通常包含 /s? 或者 /search?。

找到请求后，怎么看它发了什么？

看右侧的 Inspectors（检查器）：

1. 看请求（Request - 上半部分）

不要直接看 Raw（原始数据太难看），点击 WebForms 选项卡。

  • WebForms：它会把杂乱的 URL 参数格式化成表格。
  • 找重点：会看到一个名为 wd (word) 或者 q 的字段，后面的值就是 HelloWorld。
  • 结论：这就证明了，前端确实把搜索词通过 wd 这个参数发给了后端。

2. 看响应（Response - 下半部分）

  • 如果返回的是网页（百度搜索结果页）：点击 WebView（可以预览网页效果）或者 TextView（看 HTML 源码）。
  • 如果返回的是数据（通常是接口）：点击 JSON。

常见问题

乱码

点开某个请求，右侧 Response 区域全是乱码（看不懂的符号）。 原因：这不是 Fiddler 坏了。为了节省网络流量，服务器（百度）把数据“打包压缩”（Gzip/Brotli 格式）。浏览器会自动解压，但 Fiddler 默认展示原始的压缩数据，所以是乱码。

解决方法（两步走）：

  1. 临时解压（针对当前请求）
    • 选中那个乱码的请求。
    • 看右侧 Inspectors -> Response 区域顶部，有一个黄色的长条提示
    • 上面写着：Response is encoded and may need to be decoded before inspection. Click here to transform.
    • 点击这个黄条，乱码瞬间就会变成正常的 HTML 或 JSON 代码。
  2. 永久自动解压
    • 在 Fiddler 顶部工具栏，找到 Decode 按钮。
    • 点击它，确保它处于按下（高亮）状态。
    • 以后抓到的所有包，Fiddler 都会自动解压，再也不会有乱码了。

灰色请求

设置了过滤，但还是有很多灰色的请求。这通常是因为：Tunnel to… (443)：这是 HTTPS 建立连接的握手通道，Fiddler 必须保留它们才能解密后续内容，通常可以忽略。

与F12的区别

F12 是“显微镜”（看单个网页内部），Fiddler 是“海关检查站”（看进出电脑的所有关卡）。

如果你只是看一眼“接口返了什么”，F12 足够了；但作为测试开发，还需要做篡改、弱网、手机抓包，Fiddler 则是必选项。

1. 抓包范围（Fiddler 完胜）

  • 浏览器 F12：只能抓当前标签页。
    • 如果刷新页面，上个页面的包可能就被清空了（除非开了 Preserve log）；如果跳到了新页面，或者调用了浏览器插件的接口，F12 往往抓不到。
  • Fiddler：系统级代理（抓一切）。
    • 它能抓到 Chrome、Firefox、电脑上运行的PC端微信小程序、Outlook 邮件、某网盘客户端的包。只要是走 HTTP/HTTPS 协议的，它全都能拦下来。

场景：测试一个“PC客户端”软件（比如钉钉电脑版），F12 根本无从下手，只能靠 Fiddler。

2. 移动端测试能力

  • 浏览器 F12：无法直接抓手机 APP。
    • 虽然 F12 有个“手机模式”，那只是把浏览器窗口变窄，模拟手机分辨率，它跑的本质还是电脑网页，不是真的手机环境。
  • Fiddler：可以抓真机（手机/平板）。
    • 操作：只要手机连上代理指向电脑，你在手机 APP 上点的所有按钮，请求都会发到电脑的 Fiddler 上。

3. 篡改与 Mock 能力（Fiddler 更强更直观）

“后端接口没写好，前端怎么先测？” 或者 “如何测试支付失败的异常流程？”

  • 浏览器 F12：很难修改。
    • F12 主要用来“看”。虽然 Chrome 现在也有本地覆盖功能，但配置极其繁琐，而且修改的是静态文件，很难动态修改接口返回的 JSON 数据。
  • Fiddler：天生就是为了“改”而生的。
    • AutoResponder：可以直接把百度的 Logo 换成照片，或者把“登录成功”的接口硬改成“500 服务器爆炸”，让前端以为服务器挂了，看它会不会崩溃。

4. 协议底层细节

  • 浏览器 F12：看到的是**“浏览器处理后”**的结果。
    • 有些请求即使网络不通，F12 可能也会显示个大致情况，或者经过了浏览器缓存处理。
  • Fiddler：看到的是**“原始数据”**。
    • 它是真实的网络传输层数据。比如 HTTPS 握手过程、证书错误细节，Fiddler 能看得很清楚，而 F12 可能直接报个 (failed) 就结束了。

总结：什么时候用谁？

场景 推荐工具 原因
日常开发/调试 UI F12 查 CSS 样式、看 Console 报错、调试 JS 代码（Fiddler 做不到调试 JS）。
看接口返回 JSON F12 简单、快，不用开额外软件。
测 APP / 小程序 Fiddler F12 只有模拟器，测不了真机 APP 兼容性。
构造异常数据 (Mock) Fiddler 需要修改 Request/Response 测试前端健壮性。
模拟弱网 (慢速网络) Fiddler Fiddler 的弱网控制比 F12 更精准、可编程。
  • F12 是你的左手：用来快速定位是不是前端 JS 报错了。
  • Fiddler 是你的右手：用来分析协议、抓手机包、做破坏性测试（Mock）。

模拟请求与模拟响应

Mock（挡板测试）与接口重放。

  1. 模拟请求：后端接口修好了，想快速验证一下，但不想去页面上填那十几个输入框，这时候用 Fiddler 重放。
  2. 模拟响应：想测 APP 在“服务器挂了（500）”或“数据为空”时的表现，但服务器现在好好的，这时候用 Fiddler 篡改。

第一部分：模拟请求（Composer —— 接口重放）

这相当于 Fiddler 内置了一个简易版的 Postman。

场景：刚才在百度搜索了 helloworld，现在想模拟搜索 fiddler，但不想打开浏览器去输入，直接发包。

操作步骤：

  1. 找到原请求：在左侧会话列表中，找到刚才搜索 helloworld 的那条记录（URL 里含 /s?wd=...）。
  2. 拖拽（Drag & Drop）：
    • 鼠标左键按住这条请求不放。
    • 直接把它拖到右侧选项卡中的 Composer（构建器）标签页里。
    • 松手后，会发现 Request Body 和 Headers 自动填好了！
  3. 修改参数：
    • 在 Composer 的 Request Body 区域（如果是 GET 请求则在顶部的 URL 栏），找到 wd=helloworld。
    • 把它改成 wd=xiaoli。
  4. 发送（Execute）：
    • 点击右上角的 Execute 按钮。
  5. 验证：
    • 看左侧列表，会在最下方新增一条请求。
    • 点开它，看右侧 Inspectors -> WebForms，你会发现参数确实变成了 xiaoli。

测开面试点：这就叫“接口自动化”的雏形。可以用这个功能快速测试 SQL 注入（比如把参数改成 ' OR '1'='1 看服务器反不报错）。

第二部分：模拟响应（AutoResponder —— 核心 Mock）

这是 Fiddler 最强大的功能。要欺骗浏览器，让它以为服务器返回了我们指定的内容。

实战目标：将百度的“百度一下”按钮文字改掉，或者让百度首页的一张图片加载失败（模拟资源丢失）。

我们选择最直观的：让百度首页显示 404。

操作步骤：

  1. 准备工作：
    • 打开浏览器访问百度首页。
    • 在 Fiddler 左侧找到百度的请求。
  2. 建立规则：
    • 点击右侧 AutoResponder（自动响应器）选项卡。
    • 必须勾选顶部的两个框：
      • Enable rules（开启规则）。
      • Unmatched requests passthrough（没匹配到的请求放行 —— 这步不选会导致全网断网！）。
    • 把刚才找到的 Logo 请求，直接拖入 AutoResponder 的空白列表中。
  3. 修改响应（篡改）：
    • 在 AutoResponder 列表里选中刚才那条规则。
    • 看下方的 Rule Editor（规则编辑器）：
      • 第一行（匹配条件）：保持不变（是 Logo 的 URL）。
      • 第二行（响应行为）：点击下拉框，选择 404_Plain.dat（或者 500_Plain.dat）。
    • 点击右边的 Save。
  4. 见证奇迹：
    • 回到浏览器。
    • 强制刷新（这是关键）：按 Ctrl + F5（防止浏览器用本地缓存，强制它重新去问 Fiddler）。
    • 你会发现百度首页变成404了。

进阶玩法（替换文件）：

在第3步的下拉框里，选择最后一项 Find a file…，然后从你电脑桌面选一张搞怪的图片（比如表情包）。保存后刷新百度，百度的 Logo 就会变成你的表情包！

“如何测试系统的健壮性（Robustness）”

“使用 Fiddler 的 AutoResponder 功能进行 Mock 测试。

比如在测试转账功能时，前端发起了请求，通过 Fiddler 拦截响应，

  1. 将状态码改为 500，看 APP 是否会崩溃，还是会优雅地提示‘服务器繁忙’。
  2. 将返回的数据篡改为超长字符串或 Null，测试前端是否有数据校验机制。”

弱网测试

即模仿网速慢的情况

第一阶段：一键开启“拨号上网”模式

Fiddler 内置了一个最简单的开关，能瞬间把百兆光纤变成 90 年代的电话拨号网速。

  1. 开启：
    • 点击顶部菜单栏 Rules -> Performance。
    • 勾选 Simulate Modem Speeds（模拟调制解调器速度）。
  2. 体验：
    • 现在去刷新一下百度或者学校官网。
    • 你会发现图片是一行一行慢慢刷出来的，文字出现得很慢。
    • 注意：这个模式是全局的！开启后，电脑上所有软件（包括微信、视频）都会变慢。
  3. 关闭：
    • 测试完一定要记得去把那个勾去掉！

第二阶段：进阶——自定义网速（精准模拟 3G/4G）

问：“Fiddler 默认的弱网速度是多少？怎么修改成我想要的 3G 速度？”

默认的 Simulate Modem Speeds 实在是太慢了（它模拟的是老式 Modem），在现代测试中，通常需要自定义数值。

需要修改 Fiddler 的脚本文件（Script）。只需要改两个数字。

操作步骤：

  1. 打开脚本编辑器：

    • 点击菜单栏 Rules -> Customize Rules...。
    • 这会弹出一个编辑器窗口（Fiddler ScriptEditor），里面满是代码。

  2. 查找代码：

    • 按 Ctrl + F，输入搜索关键字：m_SimulateModem。
    • 点击“Find Next”：

  3. 读懂参数（面试考点）：

    • request-trickle-delay: 上传延迟。这里的 300 意思是：每上传 1KB 数据，强制延迟 300 毫秒。
    • response-trickle-delay: 下载延迟。这里的 150 意思是：每下载 1KB 数据，强制延迟 150 毫秒。

    数值越大 = 网速越慢。

  4. 修改数值（模拟不同网络）：

    可以根据以下参考值修改这两个数字（保存即生效）：

    网络类型 上传延迟 (request) 下载延迟 (response) 备注
    2G (极慢) 1000 800 几乎打不开图片
    3G (卡顿) 300 150 默认设置，适合测超时
    弱 4G (抖动) 100 50 偶尔卡顿，模拟信号差

  5. 实测效果：

    • 确保 Rules -> Performance -> Simulate Modem Speeds 还是勾选状态。
    • 去刷新网页。会发现即使是一张小图片，都要加载好几秒！

第三阶段：弱网测试要“看”什么？

作为测试开发，开启弱网后，要重点观察以下几个现象：

  1. 超时处理（Timeout）：
    • 如果网速慢到一定程度，APP 应该弹出提示“网络连接超时，请重试”，而不是一直白屏或崩溃。
  2. 重复请求（Duplicate Requests）：
    • 在弱网下，快速点击“提交”按钮，Fiddler 里会不会出现多条相同的请求？（如果有，就是严重 Bug，后端没做幂等性校验，前端没做防抖）。
  3. UI 异常：
    • 图片没加载出来时，是否有默认的占位图（骨架屏）？还是界面排版直接乱了？

前端性能测试

Fiddler 的性能分析主要关注网络层面的性能，也就是：“数据从服务器跑到浏览器，到底花了多久？堵在哪了？”

主要使用两个核心面板：Statistics（统计） 和 Timeline（时间轴）。

Statistics（统计面板）

这批请求一共多大？花了多久？谁最占网速？

实操步骤：

  1. 清空 Fiddler：Ctrl + X。
  2. 打开一个网页：比如访问百度或者学校官网。
  3. 停止抓包：页面加载完按 F12，防止新数据干扰。
  4. 选中所有请求：在左侧列表中，Ctrl + A 全选（或者选中你想分析的那几十条请求）。
  5. 查看右侧面板：点击 Statistics 选项卡。

你需要关注的核心数据（面试考点）：

  1. Bytes Sent / Received (流量大小)
    • 下面会显示 ESTIMATED WORLDWIDE LATENCY（预估全球延迟）。
    • 分析点：如果 Bytes Received 居然有 10MB，那网页肯定慢。检查是不是图片没压缩，或者引入了不必要的超大 JS 库。
  2. Response Counts (请求数量)
    • 分析点：如果一个简单的页面发了 200 个请求，说明前端资源太碎了。建议前端做资源合并（雪碧图、JS合并），减少 HTTP 连接开销。
  3. Show Chart (饼图)
    • 点击面板底部的 Show Chart 链接。
    • 它会弹出一个饼图，告诉流量都去哪了（图片占多少、JS 占多少）。

Timeline（时间轴）

这是最专业的视图，用于分析并发情况和阻塞问题。

实操步骤：

  1. 在左侧列表中，选中刚才那几十条请求（Ctrl + A）。
  2. 点击右侧的 Timeline 选项卡（或者右键 -> Timeline）。
  3. 看到一张色彩斑斓的瀑布流图（Waterfall Chart）。

如何看懂这张图（测试开发必修）：

这张图横轴是时间，纵轴是请求。

  1. 看“长条”的颜色与长度：
    • 绿色条（下载中）：越长说明文件越大，或者网速越慢。
    • 黑色竖线（连接复用）：这代表浏览器复用了之前的 TCP 连接，是好事（说明 HTTP Keep-Alive 生效）。
    • 红色（错误）：如果是红色，说明请求报错（404/500），必须修。
  2. 看“阶梯”形状（并发 vs 阻塞）：
    • 好的性能：图形应该是**“高耸的”**，很多条子同时开始（并行下载）。
    • 差的性能：图形是**“细长的阶梯状”**，上一个没下完，下一个就不开始（串行阻塞）。
    • 测开建议：如果看到明显的阶梯状，要建议前端检查代码，是不是 JS 放在了头部阻塞了页面渲染，或者没有开启 HTTP/2。
  3. 看“空白”间隙（服务器延迟）：
    • 如果条子前面有一大段灰色或空白，说明浏览器发了请求，但服务器迟迟不响应。
    • 结论：这是后端的问题（数据库查询慢、逻辑处理慢），不是前端的问题。甩锅给后端开发！

单请求详情（TTFB 分析）

有时候整体很快，就是某一个接口卡住了。需要分析这一个接口慢在哪。

  1. 双击左侧那个很慢的请求。
  2. 点击右侧 Inspectors -> Statistics（注意是 Inspectors 里的子标签，或者直接看 Connection 部分）。
  3. 查看 Timings 区域的数据。

关键指标（面试必问）：

  • DNS Lookup：DNS 解析时间。如果这里超过 100ms，说明域名解析有问题。
  • TCP/IP Connect：建立连接时间（三次握手）。如果大，说明网络延迟高（比如服务器在美国）。
  • Time to First Byte (TTFB)：首字节时间。
    • 含义：从发送请求到收到服务器返回的第一个字节，等了多久。
    • 判定：这是衡量后端处理速度的最核心指标。
    • 标准：如果 TTFB > 500ms，后端接口性能一定有问题（需要优化 SQL 或代码逻辑）。

手机抓包

手机（安卓或苹果均可）和电脑连接在同一个 Wi-Fi 下。电脑端设置 -> 手机端代理 -> 安装证书

电脑端设置

  1. 打开设置：
    • 菜单栏 Tools -> Options。
  2. 切换选项卡：
    • 点击 Connections 选项卡。
  3. 关键勾选：
    • 勾选 Allow remote computers to connect（允许远程计算机连接）。
    • 注意：Fiddler listening on port 默认是 8888，不用改，记下来就行。
  4. 重启 Fiddler（非常重要）：
    • 配置修改后，必须重启 Fiddler 才会生效！请关闭并重新打开它。
  5. 查看本机 IP：
    • 将鼠标悬停在 Fiddler 窗口工具栏最右侧的 Online 按钮（或者一个小电脑图标）上。
    • 它会浮现一个小黑条，显示你的 IP 地址。通常是 192.168.x.x。记下这个 IP，我们暂且叫它 电脑IP。

手机端设置

让手机的流量不直接去互联网，而是先绕道去电脑的 Fiddler 走一圈。

  1. 打开 Wi-Fi 设置：
    • 进入手机 设置 -> Wi-Fi (WLAN)。
    • 找到当前连接的那个 Wi-Fi（必须和电脑是同一个）。
  2. 修改网络配置：
    • Android：通常是长按 Wi-Fi 名称 -> 修改网络 -> 勾选 高级选项。
    • iOS：点击 Wi-Fi 名称右边的蓝色 i 图标 -> 滑到最下面找到 配置代理。
  3. 设置手动代理：
    • 代理模式选择：手动 (Manual)。
    • 主机名 (Server/Host)：输入刚才查到的 电脑IP（例如 192.168.1.5）。
    • 端口 (Port)：输入 8888。
  4. 保存。

🚨 验证时刻：

此时，请在手机上随便打开一个浏览器（Safari 或 Chrome），访问 http://www.baidu.com。

  • 如果能打开，且 Fiddler 列表里瞬间跳出一堆百度的请求 -> 连接成功！
  • 如果打不开（网页无法访问），通常是因为电脑的防火墙挡住了。
    • 解决：暂时关闭电脑防火墙，或者允许 Fiddler 通过防火墙。

安装手机证书 (解密 HTTPS 必做)

如果不装证书，你抓到的 APP 包全是灰色的锁头，或者显示 Tunnel to...，看不到内容。

  1. 下载证书：
    • 保持代理开启状态。
    • 打开手机浏览器（推荐系统自带的 Safari 或 Chrome）。
    • 在地址栏输入：http://电脑IP:8888 （例如 http://192.168.1.5:8888）。
    • 会看到一个简陋的 Fiddler Echo Service 页面。
    • 点击最下方的粗体链接：FiddlerRoot certificate。
    • 手机会提示下载，点击“允许”或“下载”。
  2. 安装与信任证书（iOS 和 Android 略有不同）：
    • 🤖 Android (安卓)：
      • 打开下载的文件，系统会问证书名称，随便填（如 fiddler），凭据用途选“VPN和应用”或“WLAN”，点击确定安装。
      • 注意：安卓 7.0 以上版本安全性很高，部分 APP 默认不信任用户安装的证书（只信任系统证书）。如果你发现浏览器能抓包但 APP 抓不到，这是正常的。“对于安卓 7.0+，我们需要用 Xposed 模块 JustTrustMe 来绕过证书校验，或者让开发打一个允许 Debug 证书的测试包。”
    • 🍎 iOS (苹果) —— 必须做两步！
      1. 安装：去 设置 -> 已下载描述文件 -> 点击安装 Fiddler 证书。
      2. 信任 (关键)：去 设置 -> 通用 -> 关于本机 -> 拉到最下面 证书信任设置 -> 找到 DO_NOT_TRUST_FiddlerRoot -> 开启开关（变绿）。

第四步：实战抓包

手机已经完全被 Fiddler 监控了。

  1. 打开手机上的知乎、小红书或者学校官网 APP。
  2. 随便刷一刷。
  3. 看电脑上的 Fiddler 屏幕。
    • 能看到大量的请求疯狂滚动。
    • 找那些 Host 是 api.zhihu.com 或者类似的请求。
    • 点开看 JSON，能看到刚才刷到的文章标题！

当结束抓包学习后，一定要把手机的代理关掉！

否则当电脑关机或 Fiddler 关闭后，手机就会断网（因为它还在傻傻地找代理）。

  • 手机 Wi-Fi 设置 -> 代理 -> 改回 “无”。
