Mysql学习

Mysql基础概念

MySQL 是一个关系型数据库管理系统（RDBMS）。

  • 关系型：数据以“表”（Table）的形式存储，表与表之间有关联（Relation）。

  • C/S 架构：MySQL 分为服务端（Server）和客户端（Client）。

    • 服务端：负责存储数据、处理查询（就在你的服务器或本地后台运行）。
    • 客户端：负责发送指令（你用的 Navicat、DBeaver，或者 Python 代码，本质都是客户端）。

  • 核心层级结构

  • Database（数据库）：相当于整个 Excel 文件（比如 Project_A.xlsx）。

  • Table（表）：相当于 Excel 中的一个 Sheet（比如 User_Sheet）。

  • Column（列/字段）：表头，定义数据的属性（如 username, password）。

  • Row（行/记录）：具体的每一行数据。

SQL 语言四大分类（DDL, DML, DQL, DCL）

1. DDL (Data Definition Language) —— 数据定义语言

作用：用来定义数据库对象（库、表、列）。也就是搭建架子。

  • 测开场景：自动化脚本运行前，初始化测试环境（建表），或测试结束后清理环境（删表）。
  • 关键词：CREATE, DROP, ALTER, TRUNCATE

  • 测开场景：构造测试数据（Insert），修改订单状态以模拟支付成功（Update），清理脏数据（Delete）。

问：DELETE、TRUNCATE 和 DROP 都是删除，有什么区别？

  1. 所属分类不同：
    • DELETE 是 DML（操作数据）。
    • TRUNCATE 和 DROP 是 DDL（定义结构）。
  2. 回滚（事务安全）：
    • DELETE 可以回滚（如果开启了事务，误删可以救回来）。
    • TRUNCATE 和 DROP 不可回滚（删了就真没了）。
  3. 删除力度：
    • DELETE：删数据，保留表结构，速度慢（一行行删）。
    • TRUNCATE：删数据，保留表结构，速度快（直接重置表）。
    • DROP：连表结构一起炸掉。

约束 (Constraints)

在测开工作中，前端页面限制了输入，但后端接口不一定限制了。直接绕过前端调接口是测开发现 Bug 的常见手段。你需要熟知数据库层面的防线。

五大核心约束：

约束类型 关键字 测开应用场景 (Test Case)
主键 PRIMARY KEY 唯一标识。测试：插入重复 ID，数据库应报错。
非空 NOT NULL 必填项。测试：传入 NULL 或不传该字段，验证接口是否拦截。
唯一 UNIQUE 手机号/邮箱注册。测试：注册已存在的手机号，验证是否提示“已存在”。
默认 DEFAULT 状态字段。测试：创建订单时不传状态，验证是否默认为“待支付”。
外键 FOREIGN KEY 数据一致性。测试：删除父表数据（如班级），验证子表（学生）数据是否被处理。

常用函数 (Functions)

1. 聚合函数 (Aggregation)

用于统计。注意：聚合函数通常配合 GROUP BY 使用，且不能用在 WHERE 后面。

  • COUNT(*) / COUNT(列名)：统计行数。
  • SUM(), AVG(), MAX(), MIN()：求和、平均、最大、最小。

2. 字符串函数

  • CONCAT(s1, s2): 拼接字符串。场景：构造全名 CONCAT(last_name, first_name)。
  • SUBSTRING(str, pos, len): 截取。

3. 日期函数 (重点)

测开经常要查询“最近 7 天的日志”或者“过期的订单”。

  • NOW(): 当前时间。
  • DATE_ADD(date, INTERVAL 1 DAY): 日期加法。
  • DATEDIFF(date1, date2): 计算两个日期相差的天数。

核心 DQL 进阶

问：SQL 的执行顺序是什么？

顺序：

  1. FROM (先确定从哪张表找)
  2. JOIN / ON (如果有连接，先拼接数据)
  3. WHERE (分组前过滤数据) ⬅️ 关键点：这里不能用聚合函数
  4. GROUP BY (数据分组)
  5. HAVING (分组后过滤数据) ⬅️ 关键点：这里可以用聚合函数，如 COUNT(id) > 5
  6. SELECT (最后选择显示哪些列)
  7. DISTINCT (去重)
  8. ORDER BY (排序)
  9. LIMIT (分页)

4.不相关子查询

核心特点：子查询可以独立运行。它不依赖外部查询的任何数据。

执行流程：

  1. 先执行子查询（只执行一次）。
  2. 拿到子查询的结果。
  3. 再执行外部查询。

场景： 查询工资高于所有员工平均工资的员工名字。

5.相关子查询

核心特点：子查询不能独立运行。子查询内部引用了外部查询表中的字段。 执行流程：循环执行（Loop）。

  1. 外部查询取出一行数据（Row 1）。
  2. 将 Row 1 的某个值（比如 dept_id）传给子查询。
  3. 子查询运行，算出结果。
  4. 外部查询根据结果判断是否保留 Row 1。
  5. 外部查询取出下一行（Row 2）……重复上述过程。

窗口函数

传统的 GROUP BY 有一个巨大的痛点：它会把多行数据“坍缩”成一行。

有一张学生成绩表。

  • 需求 A：算出每个班级的平均分。
    • ✅ 用 GROUP BY 没问题，算出每个班一行数据。
  • 需求 B：算出每个班级排名第一的学生，并列出他的名字和分数。
    • ❌ 用 GROUP BY 就很尴尬了。因为一旦 GROUP BY 班级，你就丢失了“学生名字”这个细节，只剩下聚合后的数据了。

窗口函数的作用：它可以在不减少原表行数的情况下，对每一行数据进行“局部聚合”或“排名”。

简单说：它能让你同时看到“班级平均分”和“学生个人成绩”在同一行里。

核心语法公式

窗口函数的标志就是关键字 OVER。

  • PARTITION BY：相当于 GROUP BY，把数据分成一个个小窗口（比如按“班级”分）。
  • ORDER BY：在每个小窗口内部怎么排（比如按“分数”倒序）。
  • <窗口函数>：你想在这个窗口里算什么？（排名？求和？平均？）

最常用的三剑客：排名函数

这是面试最爱问的。假设有 4 个学生的成绩：100, 90, 90, 80。我们看看三个不同的函数算出来的排名有什么区别：

  1. ROW_NUMBER() —— 不管重名，硬排
    • 结果：1, 2, 3, 4
    • 特点：即使分数一样，也会强制分出先后（通常按记录读取顺序）。
    • 场景：分页查询，或者只需要取出任意一条重复数据时。
  2. RANK() —— 并列跳号
    • 结果：1, 2, 2, 4 (注意没有第 3 名)
    • 特点：分数一样并列，但后面的名次会跳过。
    • 场景：像奥运会奖牌榜，如果有两个银牌，就没有铜牌了。
  3. DENSE_RANK() —— 并列不跳号
    • 结果：1, 2, 2, 3
    • 特点：分数一样并列，后面的名次紧接着排。
    • 场景：面试最常用！ 比如“找出每门课前三名的学生”，通常希望即使有人并列，也不要少人。

LAG()和LEAD()

  • LAG(col, n): 向前看 n 行。
  • LEAD(col, n): 向后看 n 行.

场景：检查日志表中，有没有哪次操作的时间间隔异常（例如两次点击间隔小于 1 秒）。你需要拿当前行的时间减去上一行的时间。

事务

概念

事务就是一组 SQL 语句的集合。这组语句要么全部成功，要么全部失败（回滚到原样），不能只执行一半

在 MySQL 中，默认情况下每条 SQL 都是自动提交（Auto Commit）的。要使用事务，需要手动“告诉”数据库。

  1. 开启事务：START TRANSACTION; 或 BEGIN;
    • 潜台词：数据库兄弟，接下来我说的话先别急着存硬盘，先拿个小本本记下来。
  2. 提交事务：COMMIT;
    • 潜台词：没问题了，把刚才的操作永久写入硬盘吧。
  3. 回滚事务：ROLLBACK;
    • 潜台词：我草刚才的操作搞错了/出错了，撤销！全部恢复成 BEGIN 之前的样子。

事务的四大特性 (ACID)

  1. A - Atomicity (原子性)
    • 解释：要么全做，要么全不做。事务是不可分割的最小单位。
    • 原理：靠 Undo Log 日志实现（记录了怎么撤销你的操作）。
  2. C - Consistency (一致性)
    • 解释：事务前后，数据的完整性没有被破坏。
    • 例子：转账不管成不成功，A 和 B 的钱加起来的总数是不变的。
  3. I - Isolation (隔离性)
    • 解释：多个事务同时进行时（并发），互不干扰。
  4. D - Durability (持久性)
    • 解释：事务一旦 COMMIT 提交，数据就永久保存在磁盘上了，即使数据库崩溃重启，数据也不会丢。
    • 原理：靠 Redo Log 日志实现。

并发带来的问题（隔离级别）

当两个用户（两个事务）同时操作同一张表时，会发生什么奇葩现象？

假设表里现有存款：1000 元。

1. 脏读 (Dirty Read) —— 最严重的 Bug

  • 场景：事务 A 正在把余额改成 2000，还没提交。事务 B 居然读到了 2000。结果事务 A 回滚了（改回 1000）。
  • 后果：事务 B 拿着根本不存在的“2000 元”去操作了。

2. 不可重复读 (Non-repeatable Read)

  • 场景：
    • 事务 A 第一次查，余额 1000。
    • 此时，事务 B 冲进来，把余额改成 200 (并且提交了)。
    • 事务 A 还没结束，又查了一次，余额变成 200 了。
  • 问题：同一个事务里，两次查询结果不一样，像见了鬼一样。

3. 幻读 (Phantom Read)

  • 场景：
    • 事务 A 查了一下，“咦，工资表中没有‘李同学’这条记录”。
    • 此时，事务 B 冲进来，插入了一条“李同学”的记录并提交。
    • 事务 A 准备插入“李同学”，数据库报错：“主键冲突！李同学已存在”。
    • 事务 A 傻了：“我刚才查明明没有啊？怎么像出了幻觉一样？”
  • 区别：不可重复读重点在于修改（值变了），幻读重点在于新增/删除（行数变了）。

五、 MySQL 的解决方案：事务隔离级别

为了解决上面三个问题，数据库提供了四种隔离级别（从低到高）：

隔离级别 脏读 不可重复读 幻读 性能 备注
Read Uncommitted (读未提交) ✅发生 ✅发生 ✅发生 极快 没人用，不安全
Read Committed (RC) (读已提交) ❌解决 ✅发生 ✅发生 快 Oracle/Postgres 默认
Repeatable Read (RR) (可重复读) ❌解决 ❌解决 ❌大部分解决 一般 MySQL 默认级别
Serializable (串行化) ❌解决 ❌解决 ❌解决 极慢 也就是排队做，完全没并发

💡 面试官坑点：

问：MySQL 默认的隔离级别是什么？

答：Repeatable Read (RR)。

问：那 Oracle 呢？

答：Oracle 默认是 Read Committed (RC)。

存储引擎

一、 什么是存储引擎？

想象开了一家银行（MySQL Server）。

  • Server 层（老板）：负责接待客户（连接器）、听懂客户需求（分析器）、制定存取款计划（优化器）。这层决定了 SQL 语法怎么写。
  • 存储引擎层（柜员）：老板不干苦力，真正负责把钱存进保险柜或者从保险柜拿钱的人是柜员。

柜员是**可插拔（Pluggable）**的。可以根据业务需求，随时换掉柜员。

  • 想存钱安全、不出错？用 InnoDB（细心严谨的柜员）。
  • 想存取速度极快、不关心记账细节？用 MyISAM（动作飞快的柜员）。

二、 两大主角：InnoDB vs MyISAM

1. InnoDB (默认引擎，老大哥)

从 MySQL 5.5 版本开始，它就是默认引擎了。

  • 特点：稳重、安全。
  • 必杀技：
    1. 支持事务 (ACID)：支持回滚，崩了能恢复。
    2. 行级锁 (Row Lock)：你改你的行，我改我的行，互不干扰（并发高）。
    3. 支持外键：保证数据完整性。
  • 适用场景：几乎所有现代业务（支付、电商、用户系统）。测开工作中遇到的表 95% 都是 InnoDB。

2. MyISAM (老前辈，已逐渐淘汰)

MySQL 5.5 之前的默认引擎。

  • 特点：简单、快（某些特定场景下）。
  • 必杀技：
    1. 不支持事务：崩了数据就丢了，没法回滚。
    2. 表级锁 (Table Lock)：一个人写数据，整张表锁死，其他人全得等着（并发极差）。
    3. 文件小：存储结构简单，省磁盘空间。
  • 适用场景：极少写的纯读业务（比如纯展示的日志归档、旧版博客文章表）。

三、 面试高频 PK 表（背下来）

这是测开面试中区分度很高的考点，特别是关于 Count(*) 和锁的区别。

特性 InnoDB (推荐) MyISAM (过时) 测开面试解读
事务 (Transaction) ✅ 支持 ❌ 不支持 只要涉及钱/核心数据，必须选 InnoDB。
锁机制 (Locking) 行锁 (Row) 表锁 (Table) MyISAM 在高并发写入时会死锁或排长队。
外键 (Foreign Key) ✅ 支持 ❌ 不支持 只有 InnoDB 能利用数据库层面做外键约束测试。
Count(*) 效率 慢 (要一行行扫描) 极快 (内部存了一个计数器) 如果业务需要频繁查总数且对精确度要求不高，MyISAM 有优势。
崩溃恢复 ✅ 自动恢复 (Redo Log) ❌ 容易损坏 突然断电测试：InnoDB 数据还在，MyISAM 可能文件损坏。

💡 面试官陷阱题：

问：SELECT COUNT(*) FROM table 哪个引擎快？

答：如果不加 WHERE 条件，MyISAM 快。因为它内部直接存了总行数，拿出来就是 O(1)。而 InnoDB 因为有多版本并发控制（MVCC），它不确定此刻有多少行对当前事务可见，所以必须一行行去数。

四、 测开实战中的应用

场景 1：并发性能测试瓶颈分析

  • 现象：你在做压测，发现并发上不去，大量请求超时。
  • 排查：你去看数据库监控，发现数据库锁等待严重。
  • 原因：开发人员建表时偷懒或者复制粘贴，用了默认的 MyISAM 引擎。结果一个用户修改数据，整张表锁住了，导致 TPS（每秒事务数）极低。
  • 你的价值：指出这个问题，建议改为 InnoDB，性能瞬间提升。

场景 2：脏数据清理

  • 操作：我们之前说的 BEGIN … ROLLBACK 清理测试数据大法。
  • 前提：表必须是 InnoDB。如果你在一个 MyISAM 表上执行 ROLLBACK，MySQL 不会报错，但数据根本撤销不回来（因为不支持事务）。这会让你的自动化脚本变得不可靠。

索引

底层原理

一、 什么是索引？

索引就是一本“字典的目录”。

  • 没有索引（全表扫描）：

    想在一本 1000 页的乱序书里找一个名字LiHongXiang。只能从第 1 页翻到第 1000 页。如果运气不好在最后一页，就要翻 1000 次（IO 操作）。

    • 效率：O(N)

  • 有索引：

    先查目录，拼音 L -> Li -> 找到页码 520。你直接翻到 520 页。可能只需要翻 3 次。

    • 效率：O(\log N)

  • 好处：查询（SELECT）快得飞起。

  • 坏处：写入（INSERT/UPDATE/DELETE）变慢。

    • 为什么？ 因为每存一条数据，不仅要写数据表，还要去修改目录（重新排序）。
    • 所以如果发现批量插入测试数据特别慢，第一时间检查是不是表上的索引建太多了。

二、 索引的数据结构：为什么是 B+ 树？

MySQL 的 InnoDB 引擎，索引底层使用的是 B+ 树。

为什么不用哈希表？为什么不用二叉树？

  1. 哈希表 (Hash)：
    • 虽然查 WHERE id = 1 极快（O(1)），但它不支持范围查询。
    • 想查 WHERE score > 60，哈希表就瞎了，因为它存的数据是无序散列的。
  2. 二叉树 (Binary Tree)：
    • 如果数据很多，树会变得非常高（瘦高）。
    • 每一层节点都需要一次磁盘 IO。树太高，IO 次数多，速度就慢。

B+ 树的超能力：它是一种多路平衡查找树。简单说，它长得**“矮胖”**。

  • 矮胖：即使存千万级的数据，树的高度通常也只有 3 层左右。这意味着查任何数据只需要 3 次磁盘 IO。
  • 叶子节点手拉手：注意看图的最下面，所有的数据都存在叶子节点（Leaf Nodes），而且叶子之间用双向链表连起来了。
    • 这就是为什么 MySQL 范围查询（BETWEEN, >, <）特别快的原因——找到起点，顺着链表往下拿就行了。

三、 InnoDB 的两大索引分类（面试核心）

在 InnoDB 引擎中，根据叶子节点存的内容不同，索引分为两类。搞懂这个，就理解了为什么有些 SQL 慢，有些快。

1. 聚簇索引 (Clustered Index) —— “主索引”

  • 地位：它是唯一的，也是最高贵的。
  • Key：主键 ID (Primary Key)。
  • Value：整行数据（所有字段都在这里）。
  • 物理存储：在 InnoDB 里，数据本身就是按聚簇索引组织的。找到了主键，就找到了这一行完整的数据。

2. 辅助索引 (Secondary Index) —— “非聚簇索引”

  • 地位：除了主键以外的索引（比如你给 name 字段建的索引）。
  • Key：你索引的那个字段（如 name）。
  • Value：主键 ID (注意！这里不存完整数据，只存主键值)。

四、 关键概念：回表 (Back to Table)

这是测开面试中判断你是否懂原理的“试金石”。

表 users 有两个索引：主键 id（聚簇），和 name（辅助）。

执行：SELECT * FROM users WHERE name = ‘Li’;

数据库经历了什么？

  1. 第一步（查辅助索引）：
    • 去 name 的 B+ 树里找 'Li'。
    • 找到了！拿到它对应的 Value —— 主键 ID（比如 id = 101）。
  2. 第二步（回表）：
    • 手里拿着 id = 101，再去 聚簇索引 的 B+ 树里找。
    • 找到了 101 对应的叶子节点。
    • 读取出这一行的完整数据（包含 age, email 等其他字段）。
  3. 返回结果。

结论：查询辅助索引通常需要扫描两棵树（除非只查 ID）。这就是所谓的“回表”。

数据结构

刚才讲存储引擎时，把引擎比作“柜员”。数据结构，就是这个柜员整理档案的方法。

数据库也是软件。MySQL 的存储引擎在把数据写到硬盘（或者内存）上时，必须选择一种方式来组织这些数据，否则数据就是一堆乱码。这种“组织方式”，就叫索引的数据结构。

不同的存储引擎，选择了不同的数据结构作为自己的“核心武器”。

  • InnoDB 引擎：
    • 选择的数据结构：B+ 树。
    • 为什么？ 因为它要支持范围查询（> 20 岁的人），也要支持排序。树结构最适合。而且 B+ 树把数据都放在叶子节点用链表连起来，扫表特别快。
  • Memory 引擎：
    • 选择的数据结构：哈希表 (Hash)。
    • 为什么？ 因为它是做临时表用的，全在内存里，追求极致的快（$O(1)$），而且通常不需要查范围（很少有人会问“缓存里大于 10 的数有哪些”）。
  • MyISAM 引擎：
    • 选择的数据结构：也是 B+ 树。
    • 但在实现上有区别：InnoDB 是把数据藏在树里（聚簇）；MyISAM 是树和数据分开放（树的叶子节点存的是数据的物理地址指针，而不是数据本身）。

为什么是 B+ 树？

  1. 二叉查找树 (BST)：
    • 原理：左边比我小，右边比我大。
    • 缺陷：如果我有 1, 2, 3, 4, 5… 存进去，树会退化成一个“链表”（一根棍子）。找 100 要翻 100 次。太慢。
  2. 平衡二叉树 (AVL) / 红黑树：
    • 原理：会自动旋转，保证左右平衡。
    • 缺陷：一个节点只存一个数。如果有 1000 万数据，树会很高（几十层）。
    • 致命伤：数据库的数据存在硬盘上。访问一层节点 = 读一次硬盘。树高 20 层 = 读 20 次硬盘 = 极慢。
  3. B 树 (B-Tree)：
    • 原理：为了把树变矮，我让一个节点能存很多个数（比如一个节点存 1000 个数）。这样树就只有 3-4 层了。
    • 缺陷：B 树的中间节点里也存了数据（Data）。这导致一个节点能存的索引（Key）变少了（被 Data 占了地盘）。
  4. B+ 树 (最终形态)：
    • 改良：
      1. 非叶子节点只存索引 (Key)，不存数据。这样一行能存更多的索引值，树就更矮更胖。
      2. 所有数据都落在叶子节点。
      3. 叶子节点之间有指针连接。
    • 结果：查任何数据都只要 2-3 次磁盘 IO。完美。

组合索引与“最左前缀原则”

如何创建出能“命中”的索引，避免全表扫描。这里有一个核心概念叫组合索引（Composite Index），以及随之而来的最左前缀原则。

有一个组合索引：INDEX idx_abc (a, b, c)。

这相当于建立了一个目录，它是先按 a 排序，a 一样时按 b 排序，b 还一样时按 c 排序。

最左前缀原则（Leftmost Prefix Principle）：查询时，MySQL 会从索引的最左边开始匹配。如果最左边的断了，后面的索引就全部失效。

把它比喻成**“爬楼梯”**：

  • 索引 (a, b, c) 就像一个三层的台阶。
  • 要想踩到 b，必须先踩到 a。
  • 要想踩到 c，必须先踩到 a 和 b。
  • 如果想跳过 a去踩 b，或者踩完 a 直接去踩 c（中间断了），索引就会失效或部分失效。

❌ 索引失效的经典案例

假设索引是 (name, age)：

  1. 全值匹配（完美命中）

    • ✅ 命中。先找 name，再找 age。

  2. 只查左边（命中）

    • ✅ 命中。只用到了 name 这一层。

  3. 跳过第一列（失效）

    • ❌ 完全失效。
    • 原因：因为 B+ 树是先按 name 排好序的。跳过 name 直接看 age，age 是乱序的（只有在 name 相同的时候 age 才有序）。这时候只能全表扫描。

  4. 模糊查询的坑（最左前缀的变体）

    • ✅ 命中。因为“Li”在索引树的最左边是排好序的。
    • ❌ 失效。因为最左边是 %（未知），树不知道从哪开始找，只能从头翻到尾。

哪些坑会导致索引失效？

测开做性能测试时，如果发现慢 SQL，通常就是踩了下面这些坑：

  1. 对索引字段做计算/函数操作
    • ❌ WHERE YEAR(create_time) = 2024
    • 原因：数据库存储的是 create_time 的原始值，不是计算后的值。你这一算，树就用不上了。
    • ✅ 优化：WHERE create_time BETWEEN '2024-01-01' AND '2024-12-31'
  2. 类型转换（隐式转换）
    • 假设 phone 字段是 VARCHAR 类型。
    • ❌ WHERE phone = 13800001234 (传了个数字)
    • 原因：MySQL 会在后台默默把字符串转成数字再比，相当于加了函数，索引失效。
    • ✅ 优化：WHERE phone = '13800001234' (传字符串)
  3. 范围查询右边的列失效
    • 索引 (a, b, c)
    • WHERE a = 1 AND b > 10 AND c = 5
    • 结果：a 用到了，b 用到了（范围），c 失效了。
    • 原因：因为 b 是范围，在 b > 10 的结果里，c 又是乱序的了。

什么时候不要建索引？

虽然索引快，但不能滥用。以下情况建了反而拖后腿：

  1. 区分度低的字段
    • 比如“性别”（男/女）、“状态”（0/1）。
    • 原因：如果一个字段只有 2 种取值，建了索引，B+ 树分叉极少。查“男”可能要扫一半的表，MySQL 优化器觉得还不如直接全表扫描快。
  2. 频繁更新的字段
    • 每次 update，都要调整树结构，性能损耗大。
  3. 数据量太小的表
    • 几百行数据，全表扫一下也就几毫秒，建索引纯属浪费空间。
