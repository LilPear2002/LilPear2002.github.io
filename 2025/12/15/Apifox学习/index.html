Apifox学习

• 接口是软件组件之间相互通信的协议和桥梁。

• 前端（如浏览器）通过调用接口来获取数据或执行操作。

• 后端处理请求并通过接口返回结果。

Query、Body、Header

API文档介绍：

Postman 的 raw 模式更灵活，适合快速调试一段随便粘贴的代码；但 Apifox 的 json 模式更规范，因为它把 JSON 当作数据模型来管理，这对于我们做自动化测试和维护接口文档非常有帮助，因为我们可以直接引用定义好的数据结构，而不是每次都复制粘贴一堆文本

断言验证

概念

第一步：找到“后置操作”

  1. 在界面下方找到 “后置操作” 标签页。
  2. 点击 “添加后置操作” 按钮。
  3. 在弹出的菜单中选择 “断言”。

第二步：配置断言规则

直接验证最核心的数据内容（JSONPath）：验证返回列表里的第一只宠物，它的 status 字段必须是 sold。

请在表格里这样填写：

  1. 名称：验证第一个宠物状态
  2. 取值来源：Response JSON
  3. 属性 (关键点)：输入 $[0].status
    • $ 代表根节点（整个返回数据）。
    • [0] 代表数组里的第一个元素（因为返回的是个列表）。
    • .status 代表这个元素的 status 字段。
  4. 断言条件：等于 (Equals)
  5. 预期值：sold

知识点（JSONPath）：这是测开必考题。面试官问“怎么提取多层嵌套的数据？”，答案就是 JSONPath。

第三步：运行验证

JSONPath

JSONPath 是一种用于从 JSON 数据结构中提取数据的查询语言，类似于 XPath 对 XML 的作用

官方说明文档：https://docs.apifox.com/doc-5725287

环境与变量

  1. “环境”：如何一键切换测试服和正式服。
  2. “变量”：如何实现“上一个接口的输出，自动变成下一个接口的输入”。

环境管理 (Environment)

有两套服务器：

  • 测试环境 (Dev/Test): http://192.168.1.100
  • 生产环境 (Prod): https://api.myapp.com

如果接口 URL 写死了 https://api.myapp.com/login，那测试的时候还得一个个改 URL，太累了。

操作练习：

  1. 寻找入口：在 Apifox 界面右上角，会看到一个显示为 “默认环境”（或者“No Environment”）的下拉框。点击它，选择 “管理环境”。
  2. 配置服务：
    • 在左侧选中 “默认环境”（或者新建一个环境叫“学习环境”）。
    • 在 “前置 URL” -> “服务” 下面填写：https://petstore.swagger.io/v2/。
    • 点击 “保存”。
  3. 修改接口：
    • 回到你刚才写的 “新增宠物” 接口。
    • 把 URL 修改为：pet/findByStatus
  4. 运行：
    • 再次点击 “发送”。Apifox 会自动把“环境里的前置 URL”和“接口路径”拼起来。

意义：以后要切换到生产环境，只需要在右上角切换环境，所有接口的地址就自动变了！

变量提取与传递 (Variable Extraction)

场景目标：

  1. 接口 A (POST)：创建一只宠物，服务器给它生成了一个 ID（比如 992211）。
  2. 中间动作：Apifox 自动把这个 992211 抓出来，存到一个叫 my_pet_id 的变量里。
  3. 接口 B (GET)：查询这只宠物，URL 自动引用这个变量。

步骤 1：在 POST 请求中“提取变量”

  1. 打开 “新增宠物” 接口 -> “修改文档”。
  2. 找到 “后置操作”，点击 “添加后置操作” -> “提取变量”。
  3. 填写提取规则（看图填空）：
    • 提取变量名：my_pet_id (这是我们自己起的名字)。
    • 变量类型：环境变量 (存在当前环境里)。
    • 取值来源：Response JSON。
    • JSONPath 表达式：$.id (表示提取返回 JSON 里的 id 字段)。
  4. 点击 “保存”。

关键验证步骤：

切换到 “运行” 界面，点击 “发送”。

  • 发送成功后，请点击右上角那个 “眼睛图标” (查看环境变量)。
  • 应该能看到里面多了一行：my_pet_id，值为刚才生成的数字。如果有值，说明提取成功！

步骤 2：创建新的 GET 接口使用变量

我们要验证刚才生成的 ID 能不能用。

  1. 新建接口：名称叫 查询刚创建的宠物。
  2. 方法：GET。
  3. URL：输入 /pet/{{my_pet_id}}。
  4. 保存 并 运行。

数据库验证

为什么要做数据库校验？

接口测试验证的是逻辑正确性（请求-响应过程是否成功）。

数据库验证验证的是数据正确性（数据是否按预期准确持久化）。

两者结合，才能构成完整的测试闭环

第一步：配置数据库连接 (全局设置)

在 Apifox 里，数据库连接是项目级别的资源。

  1. 打开设置：点击左侧菜单栏最下方的 “项目设置” (通常是个齿轮图标，或者在左侧目录的“外部资源”里)。
  2. 找到入口：在菜单中选择 “外部程序” -> “数据库连接”。
  3. 新建连接：点击右上角的 “新建”。
  4. 填写配置：
    • 类型：Apifox 支持 MySQL, SQL Server, PostgreSQL, Oracle, MongoDB, ClickHouse 等。
    • 连接名：起个名字，比如 Local_DB。
    • Host/Port/User/Password：填入数据库的真实信息。
    • 数据库名：填入具体的 Database Name。
  5. 测试连接：点击“点击测试”，如果通了会提示成功。

第二步：在接口中使用数据库 (核心逻辑)

逻辑是这样的：执行 SQL -> 拿到结果 -> 存入变量 -> 断言变量。

假设要验证“新增宠物”后，数据库里真的多了一条数据。在“新增宠物”接口的 “后置操作” 里做文章。

  1. 回到 “新增宠物” 接口 -> “修改文档”。
  2. 在 “后置操作” 点击添加 -> 选择 “数据库操作”。
  3. 配置 SQL 步骤：
    • 数据库连接：选择刚才配好的 Local_DB。
    • SQL命令：输入查询语句。
      • 这里可以用变量！
      • SELECT status FROM t_pet WHERE id = '{{my_pet_id}}'
    • 提取结果：
      • 这是关键。Apifox 不会直接拿 SQL 结果去断言，而是先把结果存给一个变量。
      • 变量名：db_status_verify
      • JSONPath：$[0].status (提取查询结果第一行的 status 字段)。

第三步：断言数据库结果

现在，变量 db_status_verify 里存储的就是数据库里的真实值。我们要拿它和我们期望的值做对比。

  1. 继续在 “后置操作” 里，点击添加 -> “断言”。
  2. 配置断言：
    • 对象：变量 (注意：这里不再选 Response JSON 了)。
    • 变量名：db_status_verify (刚才存 SQL 结果的那个变量)。
    • 条件：等于。
    • 预期值：sold。

流程闭环了： 接口返回成功 -> 查库 -> 拿到库里数据 -> 确认库里数据也是 sold -> 测试通过。

生成API文档

Mock

Mock：就是模拟一个真实对象的行为。在接口测试中，即模拟一个真实的服务器接口，返回预设的响应数据

  • 解除阻塞：前端无需等待后端接口完成，即可对接调试；测试可提前编写测试用例。
  • 模拟异常：轻松模拟各种异常场景（如服务器错误500、网络超时、 返回空数据），测试前端兼容性和鲁棒性。
  • 隔离测试：当某个下游接口不稳定时，可Mock它返回稳定数据，从而隔离并测试自身服务的逻辑。

为什么需要Mock？

用一个**“餐厅”**的比喻来给你解释，你就全懂了。

1. 谁在用这个 Mock 结果？

答案：主要是前端开发人员（做网页/APP的人），其次是测试人员。

想象一下：

  • 后端（厨师）：负责做菜（写代码，查数据库，返回真实数据）。
  • 前端（服务员/装修工）：负责把菜端上来，摆盘，展示给客人看（把数据显示在网页上）。
  • 接口文档（菜单）：规定了这道菜叫什么，有哪些配料（字段）。

场景一：厨师还没来上班（后端没开发完）

项目刚开始，后端还在设计数据库，代码一行都没写。但是前端不能干等着啊！前端需要把网页的“骨架”搭好，把“头像”、“用户名”这些位置留出来。

  • 如果没有 Mock：前端只能瞎写，或者写死数据（硬编码），等后端写好了再改代码，效率极低。
  • 有了 Mock：在 Apifox 里定义好字段，Apifox 就充当了一个 “塑料食物模型机器”。前端直接请求 Mock 地址，拿到了假的“塑料菜”（Mock数据），先把盘子摆好。等真的厨师（后端）做好了，把地址切回真实服务器就行了。

2. 为什么要在正常的接口里“改变字段/造假”？

Mock 对测开的巨大价值：模拟极端和异常场景。

例子：测试“用户被封号”的弹窗

想测试 App 上这样一个功能：如果用户被封号了，登录时会弹出一个红色的警告框。

  • 用真接口（Real Backend）：
    • 需要去联系后端改数据库，把你的账号状态改成“banned”。
    • 测试完，还得改回来，不然登录不上去了。
    • 万一把测试环境弄坏了，别人也测不了。
    • 太麻烦，太危险！
  • 用 Mock（造假）：
    • 在 Apifox 里，不需要动后端一行代码。
    • 只要在 Mock 规则里，把 status 字段强制改成 banned。
    • 然后让 App 去连 Mock 地址。
    • App 一请求，收到了 banned，立刻弹出了红色警告框。
    • 测试通过！ 验证了客户端的逻辑是正确的，而且没有骚扰后端，也没有脏数据。

3. 它和原接口有什么关系？是新接口吗？

它不是新接口，它是原接口的**“平行宇宙”**。

在 Apifox 里，同一个接口定义，有两个“门”可以进：

  1. A 门（正式环境）：http://api.myapp.com/login
    • 进去后，是真实的后端服务器在处理。
    • 它查真的数据库，返回真的结果。
  2. B 门（Mock 环境）：http://127.0.0.1:4523/m1/xxx/login (Apifox 提供的)
    • 进去后，是 Apifox 的虚拟服务器在处理。
    • 它不查数据库，它只看在文档里定义的规则（比如：让 age 随机生成 18-20）。
    • 它返回的是设定的“假数据”。

两者的关系： 它们共享同一套**“长相定义”（都有 id, name, status 字段），但“内容来源”**完全不同。

总结：Mock 到底解决了什么？

场景 不用 Mock 的痛苦 用 Mock 的爽点
后端没开发完 前端只能干等，项目延期 前端用 Mock 数据先开发，前后端并行工作
测试报错页面 很难让服务器故意报错 (500 Error) 在 Mock 里设置“期望”，直接返回 500，秒测前端容错能力
测试大量数据 手工去数据库里造 1000 条数据 Mock 设置循环规则，一秒生成 1000 条列表数据，测试分页功能

在APIfox中分为智能 Mock、自定义规则 和 高级 Mock（期望）。

智能 Mock (Smart Mock)

Apifox 会根据字段名自动猜要什么数据。

原理：Apifox 内置了一个强大的规则库，只要字段名叫 url, email, phone, city 等，它都能自动生成像模像样的假数据。

自定义 Mock 规则 (Regex/Wildcard)

有时候智能识别不够用。比如要测试国内手机号，必须是 11 位且以 1 开头，随便生成的字符串通过不了校验代码。

场景：生成真实的手机号

  1. 在响应字段里，添加一个 mobile 字段。
  2. 点击 Mock 列的输入框。
  3. 方法 A：使用内置生成器 (推荐)
    • 点击输入框，选择 “手机号”。
  4. 方法 B：使用正则表达式 (进阶)
    • 如果非常特殊的格式（比如以 138 开头的号码）。
    • 在 Mock 栏输入正则：/^138\d{8}$/。
  5. 方法 C：使用 Mock.js 语法
    • Apifox 兼容 Mock.js。输入 @integer(18, 60) 可以生成 18 到 60 岁的年龄。

第三层级：高级 Mock (期望 / Expectations) —— 测开必修

这是 Mock 的终极形态。

场景：

  • 如果我传 id=1，你要给我返回“张三”的信息。
  • 如果我传 id=999，你要给我返回“用户不存在”（404 错误）。
  • 普通 Mock 做不到这点（它只会随机），必须用 “高级 Mock”。

操作步骤：

  1. 在接口详情页，点击上方的 “高级 Mock” 标签。

  2. 点击 “新建期望”。

  3. 配置期望 1（正常情况）：

    • 名称：查询张三成功

    • 触发条件：参数 id 等于 1。

    • 响应内容：

  4. 配置期望 2（异常情况）：

    • 点击 “新建期望”。
    • 名称：查询不存在的用户
    • 触发条件：参数 id 等于 999。
    • 响应内容：
      • HTTP 状态码改为 404。
      • Body 填：{ "error": "User not found" }。

为什么这对测开很重要？

问：“你如何测试前端对‘服务器500错误’或‘超时’的处理？”

答：

“利用 Apifox 的 高级 Mock (期望) 功能。不需要去搞挂真实的服务器，只需要配置一个 Mock 期望，当传入特定参数时，Mock Server 直接返回 500 状态码。这样就能验证前端页面会不会正确弹出‘服务器繁忙’的提示，而不会白屏崩溃。”

测试场景（容器）

在“测试场景”这个容器里，可以放入 100 个接口，按顺序排列。

它的核心价值：

  1. 上下文传递：步骤 1 的结果，自动传给步骤 2（基于之前学的变量）。
  2. 流程控制：可以循环跑、等待几秒再跑、或者根据条件决定跑不跑。
  3. 独立运行：这个容器可以独立被 CI/CD 调用，也可以一键生成测试报告。

搭建基础业务流

  1. 新建场景：
    • 点击左侧菜单栏的 “自动化测试”。
    • 点击 “新建测试场景”（或者点 + 号）。
    • 命名为：宠物全生命周期测试。
  2. 导入步骤（装填容器）：
    • 在场景详情页，点击下方的 “导入步骤”。
    • 选择 “接口用例”。
    • 在弹出的列表里，勾选我们之前做好的：
      1. 新增宠物
      2. 查询已售出宠物
    • 点击 “确定”。
  3. 运行：
    • 点击右上角的 “运行”。
    • 会在右侧看到执行日志：它会自动依次执行这两个接口，如果之前的变量提取（后置操作）配置对了，它们会自动传递参数。
    • 结果：最后会生成一个总的“通过率”报告。

测开知识点（引用 vs 复制）：

Apifox 有个很好的特性：这里导入的步骤默认是 “引用” 模式。

意思是：如果你去“接口管理”里修改了原接口的参数，这个测试场景里的步骤会自动同步更新。这大大降低了维护成本！

流程控制（循环与等待）

真实的业务往往不是一根筋走到底的。

场景：用户下单后，系统处理需要时间，不能立刻查，得等 1 秒。或者想连续创建 10 只宠物压压惊。

  1. 添加等待（Think Time）：
    • 点击右侧的 “添加步骤” -> “等待”。
    • 把它拖动到“新增宠物”和“查询”之间。
    • 设置等待时间：1000 ms (1秒)。
    • （这是为了防止数据还没写入数据库，马上查会报错的常见稳健性处理）。
  2. 设置循环（Loop）：
    • 点击场景列表里的 “新增宠物” 这一行。
    • 在右侧属性栏，找到 “循环次数”。
    • 设置为 5。
    • 效果：系统会先连续执行 5 次新增，休息 1 秒，然后执行查询。

条件分支（If-Else）

这是测试脚本智能化的关键。

场景：只有当“新增宠物”成功（状态码 200）时，才去执行“查询”。如果第一步就挂了，后面没必要跑。

  1. 点击 “添加步骤” -> “条件分支” (Condition)。
  2. 配置条件：
    • 比如：判断变量 code 等于 200。
  3. 拖拽层级：
    • 把“查询宠物”这个步骤，拖进 这个“条件分支”的内部（像文件夹一样）。
  4. 逻辑：如果条件不满足，里面的步骤就会被 Skipped (跳过)。

数据驱动测试 (Data-Driven Testing)

如果想明确指定：

  • 第一次用名：Cat_A
  • 第二次用名：Cat_B
  • 第三次用名：Cat_C

操作流程：

  1. 准备数据文件 (CSV)：

    • 在电脑上新建一个 pet_data.csv 文件。

  2. 修改接口参数：

    • 回到“新增宠物”接口。
    • 把 Body 里的名字改成变量：{{pet_name}}。
    • 把状态改成变量：{{pet_status}}。

  3. 配置测试场景：

    • 回到测试场景的运行界面。
    • 打开 “测试数据” 开关。
    • 点击 “管理测试数据” -> “导入” -> 上传刚才的 CSV。
    • 关键点：会看到 Apifox 自动预览了数据，并且把 CSV 的表头（pet_name）对应到了变量。

  4. 运行：

    • 点击运行。
    • 系统会自动运行 3 次（因为 CSV 有 3 行数据）。
    • 第一次用“旺财”，第二次用“来福”…

Runner与定时任务

Runner（运行器） 解决了“批量运行”的问题。定时任务 (Scheduled Tasks) 解决了“无人值守运行”的问题。

Runner (运行器配置)

当点击“自动化测试”场景右上角的“运行”时，弹出的那个配置框就是 Runner。。

1. 核心运行参数

  • 运行环境：一定要确认好！别把脏数据写到生产环境（Prod）去了，一定要选测试环境（Test）。
  • 循环次数 (Iterations)：
    • 功能测试：通常填 1。
    • 稳定性测试：填 100 或更多。用于检测接口是不是跑着跑着就报错了（比如内存泄漏导致后面变慢）。
  • 线程数 (Threads) —— 面试考点：
    • 默认 1：串行执行。上一个跑完，下一个才开始。适合有逻辑依赖的业务（比如：注册 -> 登录）。
    • 大于 1 (比如 10)：并行执行。模拟 10 个人同时点按钮。
    • 注意：Apifox 的并发能力主要用于轻量级验证。如果问“怎么做高并发压测？”，建议回答用 JMeter 或 K6，Apifox 更多用于功能流程的自动化。
  • 遇到错误时停止：
    • 调试时勾选。
    • 正式跑回归测试时不要勾选，要看总体的成功率。

2. 运行报告 (Test Report)

跑完之后，Runner 会生成一份报告。这是你要发给领导或开发看的东西。

  • 概览：通过率（比如 95%）。
  • 失败详情：点击红色的用例，不仅能看到报错，还能看到当时发送的请求和当时返回的响应。

定时任务 (Scheduled Tasks)

场景：每天凌晨 2 点，自动把所有接口跑一遍。早上上班时，如果收到邮件说全绿，就能安心摸鱼；如果红了，马上修 Bug。

Apifox 实现定时任务主要有两种方式：

方式 A：使用 Apifox 自带的“定时任务” (最简单)

  1. 在左侧“自动化测试”栏，找到 “定时任务”。
  2. 新建任务：
    • 选择你要跑的 测试场景。
    • Crontab 表达式（核心）：这是 Linux 通用的定时规则。
      • 每天凌晨 2 点：0 0 2 * * ?
      • 每隔 10 分钟：0 0/10 * * * ?
      • Apifox 界面通常提供了可视化的选择器。
  3. 通知配置：
    • 配置钉钉/飞书机器人 webhook，或者邮件。
    • 效果：任务跑失败了，群里自动弹消息：“报警！登录接口挂了！”

方式 B：CI/CD 集成 (DevOps 标准做法 )

通过 命令行 (CLI) 来驱动测试。

  1. 安装 CLI：

    • 需要 Node.js 环境。
    • 在终端运行：npm install -g apifox-cli

  2. 获取运行命令：

    • 回到 Apifox 的测试场景页面。

    • 点击右上角的 “CI/CD” (或者叫“发布/流水线”) 按钮。

    • 配置好环境（测试环境）。

    • Apifox 会自动生成一串长长的命令，类似于：

      apifox run https://api.apifox.cn/api/v1/projects/… -r html,cli

  3. 执行与集成：

    • 在本地终端粘贴这行命令，回车，会发现测试在黑底白字的命令行里跑起来了！
    • 最终大招：把这行命令复制到 Jenkins 或 GitLab CI 的配置文件里。
    • 意义：每当开发提交代码 (Git Push)，Jenkins 自动运行这行命令。如果 Apifox 测试失败，禁止代码上线。

问：“自动化测试是怎么执行的？”

“我们的自动化测试分为两个层级：

  1. 日常调试：使用 Apifox 的 Runner 进行小批量的功能验证。
  2. 回归体系：配置 CI/CD 流水线。使用 Apifox CLI 生成了运行命令，集成了 Jenkins。
  3. 监控：设置了定时任务，每天凌晨自动全量运行一次，生成的 HTML 报告会自动发送到团队群里，确保主要业务流程没有被当天的代码搞坏。”

压力测试

Apifox 的性能测试功能不如 JMeter 那么全面（比如没有复杂的阶梯加压、分布式压测），但用来做轻量级的压力测试和冒烟压测绰绰有余。

配置并发 (设置虚拟用户)

性能测试的核心就是：模拟很多人同时点同一个按钮。

  1. 找到入口：
    • 打开 “测试场景”。
    • 点击右上角的 “运行” 按钮。
  2. 调整运行模式：
    • 默认是“自动化测试”模式（串行执行）。关注下面的参数来变成“性能测试”模式。
  3. 关键参数设置：
    • 循环次数 (Loop Count)：每个用户做几次操作？（比如填 20）。
    • 线程数 (Thread Count)：这是重点！这里代表并发人数。
      • 如果填 1，就是一个人慢慢点，不算压测。
      • 请填 10：模拟 10 个用户同时操作。
    • 总请求数：Apifox 会自动计算：10 (线程) * 20 (循环) = 200 次请求。

注意：由于 Apifox 是基于 Node.js 运行的本地程序，单机并发线程数建议不要设置过大（比如不要超过 100-200），如果需要上万并发，还是得用 JMeter。

执行压测与防呆设置

在点击“运行”之前，有两个细节

1. 固定延迟 (Think Time)

真实的 10 个用户，不会像机器一样毫秒不差地同时点击。

  • 在**“高级设置”** 里，设置 “停顿/延迟”。
  • 比如设置 500ms。
  • 意义：模拟用户思考时间。

2. 失败处理

  • “遇到错误时停止测试”：做性能测试时，千万不要勾选这个！
  • 原因：性能测试本来就是为了看服务器什么时候报错。如果有 1% 的请求失败了，我们希望看到剩下的 99% 是快还是慢，而不是立刻停下来。

分析性能报告 (核心能力)

跑完后，不要只看通过率，要看 “性能指标”。

Apifox 的报告会给你几个关键数据：

  1. 平均耗时 (Average Response Time)：
    • 比如 350ms。这代表用户平均要等多久。
    • 标准：一般来说，互联网接口要求在 200ms - 500ms 以内。
  2. 最小/最大耗时 (Min/Max)：
    • 如果最小 100ms，最大 5000ms (5秒)。
    • 说明：系统极不稳定，发生了**“抖动”**。可能是数据库锁死，或者 Java 的 GC (垃圾回收) 卡顿了。这是严重的性能隐患。
  3. 吞吐量 (TPS/RPS)：
    • 报告里可能显示 Request per Second。
    • 这代表服务器每秒能处理多少个请求。数值越大性能越好。
  4. 错误率 (Error Rate)：
    • 如果有红色报错，查看是 Timeout (超时) 还是 500 Internal Error (服务器崩了)。

导出与汇报

Apifox 允许导出 HTML 格式 的报告。

  • 点击报告右上角的 “导出”。
  • 场景：开发说“优化了代码，再测测”。
    • 跑一次，导出 Report_Before.html。
    • 再跑一次，导出 Report_After.html。
    • 对比数据：平均耗时从 800ms 降到了 300ms。

问：“既然 Apifox 能做，为什么还要招会 JMeter 的人？”

满分回答范例：

“使用 Apifox 做研发阶段的快速压测。

  1. 优势：由于平时就用 Apifox 管理接口和做自动化，脚本是现成的，改一下线程数就能跑，不需要重新写 JMeter 脚本，效率极高。适合开发自测或冒烟测试。
  2. 局限：Apifox 是单机运行，受限于本地资源，难以模拟高并发（如几万并发）或分布式压测。而且它的报告图表不如 JMeter 详细（比如没有梯形加压图）。
  3. 结论：所以，日常小并发验证用 Apifox；如果是上线前的大型全链路压测，会把 Apifox 的接口导出为 cURL 或脚本，导入到 JMeter 中去执行。”
