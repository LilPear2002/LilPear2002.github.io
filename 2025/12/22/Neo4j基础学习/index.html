Neo4j基础学习

基础概念

不同于 MySQL 的“表（Table）”和“行（Row）”，Neo4j只有 节点 (Nodes) 和 关系 (Relationships)。

  • 节点 (Node): 实体。例如：一个用户、一个测试用例、一个 Bug。
  • 关系 (Relationship): 连接实体的线，必须有方向和类型。例如：用户 -> 购买了 -> 商品。
  • 属性 (Properties): 节点和关系都可以存储数据（Key-Value对）。
  • 标签 (Label): 给节点分类。例如 :User, :Product。

简单理解： 想象在白板上画圈圈（节点）和连线（关系），这就是 Neo4j 存储数据的方式。它比 SQL 更直观，特别是在查询“朋友的朋友”这种多层关系时，性能快千倍。

  • 复杂数据构造： 构造一个“涉嫌洗钱的转账链条”测试数据，用 SQL 写起来非常痛苦，用 Neo4j 画几条线就搞定。
  • 链路分析： “修改了模块 A，会影响哪些下游服务？”，可以通过查询服务依赖图来精准确定回归测试范围。

Cypher基础

Neo4j 的查询语言叫 Cypher。它的语法非常像“画图”。

  • () 代表节点。
  • -- 代表关系。
  • -> 代表方向。
  • [] 代表关系的详细信息。

高阶知识理论

1. 逻辑与扩展：APOC (对标 Redis 的 Lua 脚本)

Redis 有 Lua 脚本来处理复杂原子逻辑，Neo4j 有 APOC (Awesome Procedures on Cypher)。

  • 理论概念： Cypher 是“声明式”语言（告诉电脑我要什么），但在做复杂数据清洗、批量导入、或者需要写类似编程语言的 if-else 循环逻辑时，Cypher 就很吃力。APOC 是 Neo4j 的一个超强插件包（现在的版本很多已内置），提供了几百个存储过程。
  • 测开场景：
    • 造海量数据： 不想写 Python 脚本去插库，可以直接用 APOC 语法：CALL apoc.periodic.iterate(...)，在数据库内部直接生成 100 万条测试数据，速度比外部脚本快 10 倍。
    • 导入数据： 直接用 APOC 读取 JSON/XML/CSV 文件并落库，不需要经过后端服务。

2. 事务与锁：ACID (对标 Redis 的 Watch/乐观锁)

Redis 的事务比较弱，但 Neo4j 是完全符合 ACID 的事务型数据库（和 MySQL 一样强）。

  • 理论概念：
    • ACID： 即使是图数据库，转账操作要么全成功，要么全失败。
    • 乐观锁 (Optimistic Locking)： Neo4j 内部也有版本控制。如果你和我都读取了节点 A，我改了它，再想改提交时就会报错（因为版本号变了）。
    • 死锁 (Deadlock)： 在高并发测试中，如果两个事务互相锁住对方需要的节点，Neo4j 会自动检测并抛出异常。
  • 测开场景：
    • 并发测试： 写测试脚本模拟 100 个人同时抢一张票（修改同一个节点的属性），需要验证 Neo4j 是否正确抛出了锁竞争异常，而不是把数据改乱了。

3. 性能调优：PROFILE 与 索引 (对标 MySQL 的 Explain)

  • 理论概念：
    • 执行计划 (Execution Plan)： 在 Cypher 语句前加 PROFILE 或 EXPLAIN，它会告诉你查询经过了哪些步骤。是“全图扫描”（AllNodesScan，类似全表扫描，性能杀手）还是走了“索引”（IndexSeek）。
    • 索引 (Index)： 图数据库也要建索引！否则查 name="张三" 也会慢死。Neo4j 支持 B-Tree 索引、全文索引（Full-text）甚至向量索引（Vector Index）。
  • 测开场景：
    • 慢查询分析： 开发写的查询太慢，你作为测开，能不能甩出一张 PROFILE 截图告诉他：“你看，你这里发生了笛卡尔积（Cartesian Product），甚至没走索引，甚至扫了 100 万个点只返回 1 个，必须优化！”

4. 架构模式：因果集群 (Causal Clustering) (对标 Redis Cluster/Sentinel)

  • 理论概念：
    • Neo4j 不只是单机版。在生产环境通常是 Core + Read Replica 架构。
    • Raft 协议： 核心节点（Core）之间通过 Raft 协议保证数据强一致性（Leader-Follower）。
    • 因果一致性 (Causal Consistency)： 这是 Neo4j 的特色。它保证“如果是你刚写的数据，你立马就能读到”（Read your own writes），但其他人可能稍有延迟。
  • 测开场景：
    • 故障演练 (Chaos Engineering)： 把 Leader 节点强行关机，测试系统是否会自动选主？业务是否中断？
    • 主从延迟： 刚在主库写了数据，马上在从库查，会不会查不到？（这是分布式系统测试的经典坑）。

5. 图算法：GDS (Graph Data Science) (Neo4j 的杀手锏)

这是 Redis 和 MySQL 完全不具备的能力。

  • 理论概念： 数据库不仅存数据，还能算数据。
    • PageRank： 算出谁是网络中最关键的节点（大V）。
    • 社群检测 (Louvain)： 自动发现哪些人是一个小团伙（反欺诈核心逻辑）。
    • 路径算法 (Dijkstra/A*)： 算两点间的最短/最优路径。
  • 测开场景：
    • 算法准确性测试： 开发上线了一个“推荐可能认识的人”功能。你怎么测？需要用 GDS 库跑一遍基准算法，验证开发算出来的结果是否符合预期。

6. Vector Search (向量搜索) —— 给 LLM 用的

  • 理论概念： Neo4j 做为向量数据库以存“向量”（Vectors）。
  • 用途： 结合 LLM（大模型）做 GraphRAG（基于图的检索增强生成）。
    • 比如：把用户的知识图谱变成向量存进去。当用户问问题时，不仅靠关键词匹配，还靠向量相似度找到关联的知识，喂给 GPT。
