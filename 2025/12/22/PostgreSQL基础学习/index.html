PostgreSQL基础学习

基础概念

PostgreSQL 的结构和 MySQL 有一个巨大的区别。

  • MySQL: 实例 -> 数据库 (Database) -> 表 (Table)
  • PostgreSQL: 实例 -> 数据库 (Database) -> 模式 (Schema) -> 表 (Table)

在 PG 中，默认的模式叫做 public。当写 SELECT * FROM users 时，其实 PG 默认帮你补全成了 SELECT * FROM public.users。在多租户系统测试中，可能会遇到不同的 Schema 代表不同的租户。

数据类型支持：PG 对 JSON 的支持非常强大（JSONB），可以直接索引 JSON 内部字段，这在测试存储复杂配置（如上述 config 字段）时非常方便，而 MySQL 相对较弱。

标准兼容性：PG 更符合 SQL 标准，支持更复杂的数据查询。

层级结构：PG 多了一层 Schema，适合做多租户隔离测试。

DQL与DML

引号与大小写

特性 MySQL PostgreSQL (PG) 测开避坑指南
字符串引号 单引号 ' 或 双引号 " 都可以 严格只认单引号 ' 写值（Value）时，永远只用单引号！ 例如 WHERE name = 'Alice'
别名/列名 用反引号 ` (键盘左上角) 用 双引号 " 只有当列名有大写或特殊字符时才需要双引号，一般可以不加。千万别在 PG 用反引号。
布尔值 0 / 1 (Tinyint) TRUE / FALSE (‘t’/‘f’) 验证接口返回字段时，MySQL 返回可能是 1，PG 返回是 true。
大小写敏感 默认不敏感 (比如查询 ‘admin’ 能查出 ‘Admin’) 默认严格敏感 查 ‘Admin’ 必须写 ‘Admin’，写 ‘admin’ 查不到。

DQL (数据查询) 的主要区别

A. 模糊查询 (大小写问题)

  • MySQL: LIKE 默认不区分大小写。
  • PG:
    • LIKE: 区分大小写。
    • ILIKE: 不区分大小写 (PostgreSQL 特有)。

B. 字符串拼接

  • MySQL: CONCAT(a, b)
  • PG: 使用双竖线 || (这是标准 SQL 写法) 或 CONCAT。

C. 分页 (Limit)

  • MySQL: 支持 LIMIT 0, 10 这种非标准写法。
  • PG: 不支持逗号写法，必须写全。

DML (增删改) 的主要区别

A. 插入/更新后的返回值 (RETURNING)

  • MySQL: 只能拿 ID，通常要再次查询才能看到某些默认值（如 create_time）。
  • PG: INSERT, UPDATE, DELETE 后面都可以接 RETURNING *，直接把受影响的行返回。

B. 冲突处理

当“如果存在则更新，不存在则插入”时：

  • MySQL: ON DUPLICATE KEY UPDATE ...
  • PG: ON CONFLICT (字段名) DO UPDATE ...
功能 MySQL 写法 PostgreSQL 写法
获取刚插入的ID SELECT LAST_INSERT_ID(); INSERT ... RETURNING id;
强制类型转换 CAST(val AS CHAR) val::text (双冒号是 PG 的语法糖，超好用)
正则查询 REGEXP ~ (例如 WHERE name ~ '^[A-Z]')
当前时间 NOW() 或 SYSDATE() NOW() 或 CURRENT_TIMESTAMP
  1. 构造数据脚本时：如果你在 Python 里写 SQL 构造测试数据，记得字符串全用单引号，否则代码在 PG 环境会直接报错。
  2. 断言验证时：如果你在做搜索功能的自动化测试，要注意 PG 的搜索默认是区分大小写的。如果开发用的 PG 但没做处理，你搜 “test” 查不到 “Test”，这就是一个 Bug（或者特性）。
  3. 使用 :: 转换：在验证数据时，PG 的 JSONB 取出来有时需要转成文本对比，会经常用到 字段::text 这种写法。

JSONB

PG 有两种 JSON 类型：

  1. JSON: 仅仅是把你的 JSON 当作纯文本存进去，读取时要重新解析，慢。
  2. JSONB (Binary): 存进去时就变成了二进制格式，去掉了空格，重新排了序。它的读取速度极快，且支持索引。

测开笔记： 只要涉及“存 JSON”，无脑选 JSONB。在测试验证时，绝大多数时候也是针对 JSONB 字段进行查询。

数组类型 (Array) 的操作

查找包含 “smoke_test” (冒烟测试) 标签的日志。不要用 LIKE，用 PG 专用的数组语法 ANY。

注意-PostgreSQL 处理 JSON 的机制：

  1. 当试图去取一个不存在的键（比如 amout）时，PG 不会报错，而是返回 NULL。
  2. 然后 NULL 被转换成浮点数（NULL::float）依然是 NULL。
  3. 最后 NULL > 50 的结果是 NULL（既不是 True 也不是 False）。

所以在写自动化校验脚本时，如果发现断言失败或者查不到数据，第一件事先检查 JSON 的 Key 有没有拼错，因为 PG 默认不会抛出异常提醒。

pgvector

PGvector 是 PostgreSQL 数据库的一个扩展，专门用于在数据库中高效地存储、检索和查询向量数据（vector embeddings）。

  1. 存储向量
    支持多种维度的向量（通常是几十到几千维），这些向量通常来自 AI 模型（如文本、图像、音频的嵌入表示）。
  2. 相似性搜索
    允许通过向量之间的距离计算（如欧氏距离、余弦相似度、内积）来查找最相似的向量。
  3. 与 SQL 无缝集成
    因为它是 PostgreSQL 的扩展，所以可以直接在 SQL 语句中执行向量操作，结合关系型数据的查询，非常灵活。
