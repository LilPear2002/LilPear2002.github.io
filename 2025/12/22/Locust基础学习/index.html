Locust基础学习

基础概念

Locust 是非常适合**测试开发（Test Dev）**岗位的工具

  1. 它是 Python 编写的：完全符合测开对代码能力的要求。
  2. Code-as-Infrastructure：测试脚本就是代码，易于维护和版本控制。
  3. 高并发能力：基于协程（Gevent），单机能模拟极高并发。

Locust 的三个核心概念：

  • User Class (用户类)：每一个虚拟用户都是这个类的一个实例。
  • Task (任务)：用户具体要做的事（比如：刷首页、下单、搜索），用 @task 装饰器标记。
  • Wait Time (等待时间)：模拟真实人类的操作间隔。

模拟一个场景：用户访问百度的首页，并且偶尔访问一下百度的“新闻”页面。

  3. 点击 “Start swarming”

跑起来后，会看到几个核心指标：

  1. RPS (Requests Per Second): 吞吐量。每秒服务器处理了多少请求。数值越大，性能越好。
  2. Failures: 失败率。如果这个不为 0，说明服务器报错了或者网络有问题。
  3. Average / Median / 95%ile (ms): 响应时间。
    • 坑点：永远不要只看 Average（平均值），因为平均值会掩盖问题。
    • 要看 95%ile (P95)：意思是 95% 的请求都在这个时间内完成了。这才是衡量用户体验的真实标准。

参数化

如果 10 个用户都搜同一个词：

  • 服务器端：可能有缓存，导致测试结果比实际快，数据失真。
  • 业务逻辑：无法覆盖数据库查询的多样性。

解决方法：建立一个“词库”，让用户每次随机挑一个词去搜。

另外，如果直接把 URL 写成动态的，比如：

  • 用户A 访问 /s?wd=apple
  • 用户B 访问 /s?wd=banana

Locust 的报告里会把这两条当成两个完全不同的接口来展示。如果有 1000 个搜索词，报告里就会有 1000 行数据，没法看。

需要做的是：告诉 Locust，“不管后面参数是什么，这些都是‘搜索接口’，请把它们合并成一行统计”。

修改 locustfile.py：

  1. 保存代码。
  2. 回到终端，必须按 Ctrl+C 停止之前的运行，然后重新输入 locust 启动（代码修改后必须重启）。
  3. 回到浏览器 http://localhost:8089，再次启动测试。

这一次，请盯着 Statistics 列表里的 Name 那一列。

  • 如果做对了：不管通过了多少请求，搜索接口那一行永远只显示 GET /s?wd=[keyword]，所有的搜索请求都被合并统计到了这一行里。
  • 如果做错了（比如忘了加 name 参数）：你会发现列表越来越长，每一行都是具体的 /s?wd=python, /s?wd=locust…

模拟登陆

如果在每一个任务里都写一遍登录，导致服务器压力爆炸（每秒登录 1000 次），还会被安全系统封号。

在 Locust 里，处理登录只需要记住两点：

  1. on_start 方法：这是一个特殊的方法。Locust 启动每个虚拟用户时，只运行一次这个方法。它是放登录代码的最佳位置。
  2. Session 自动保持：self.client 继承自 requests.Session。这意味着，只要在 on_start 里登录成功了，后续所有的 task（任务）都会自动带上 Cookie，不需要手动传 Cookie。

使用 httpbin.org（一个专门用于测试 HTTP 请求的工具网站）来模拟登录过程。

读取文件

场景：参数化登录（每个人用不同的账号登录）。

这里有两种常用的策略，需要根据业务场景选：

  1. 随机抽取（Random）：适合搜索词、商品ID。数据可以重复使用。
  2. 独占消耗（Queue）：适合登录账号。绝对不能两个虚拟用户同时用同一个账号登录，否则会被互踢。

以下用第二种：使用队列（Queue）实现账号的“独占读取”。

问：“直接把数据读到一个 list 列表里，然后用 random.choice 随机选不行吗？”

“如果是搜索关键词，可以用 list+random。

但如果是登录账号，用 random 会导致两个虚拟用户随机到了同一个账号 user1。

现实中，后登录的 user1 会把先登录的 user1 踢下线（Session 互斥），导致前一个用户的后续请求全部报错。

所以对于消耗型数据（如账号、优惠券码），必须用 Queue 做到取一个少一个，保证线程安全。”

无界面运行

在真实的生产环境中（ Linux 服务器），通常是没有图形界面的，往往需要把它集成到 Jenkins 或 GitLab CI 里的自动化流水线中，这时候必须使用命令行模式。

  • 节省资源：图形界面（Web UI）本身也会消耗电脑资源。当要模拟几万并发时，哪怕一点点资源都很宝贵。
  • 自动化集成：CI/CD 流水线（比如 Jenkins）只能运行命令，没有人去点网页上的 “Start” 按钮。

参数详解：

  • --headless: 核心开关。告诉 Locust 不要启动网页，直接在后台跑。
  • -u 10 (Users): 启动 10 个用户。
  • -r 2 (Rate): 每秒启动 2 个用户（5秒后达到满载）。
  • -t 30s (Time): 运行 30秒 后自动停止。
  • --html result.html: 测试结束后，自动生成一个漂亮的 HTML 报告文件。

输入命令后，会看到屏幕上只有简单的文字在滚动，不再有网页弹出来。

  • 它会显示 Ramping to 10 users...
  • 然后每隔几秒打印一次当前的统计数据。
  • 30秒倒计时结束后，它会自动停止，并显示 Shutting down。

去文件夹里找一找，多了一个叫 result.html 的文件。打开会看到一个包含图表、统计数据的完整报告。

“每次自动化压力测试结束后，会配置 Locust 自动导出 HTML 报告，并发送邮件给开发团队，这样大家不用跑脚本也能看到性能趋势。”

问 “为什么要用 Locust 而不用 JMeter？”

“JMeter 虽然功能强大，但它是基于线程的，并发很高时非常吃内存。而 Locust 是**基于协程（Gevent）**的，更轻量，单机能支持更高的并发。

更重要的是，Locust 是 Code-as-Infrastructure（代码即各种配置），更喜欢用 Python 代码来维护复杂的测试逻辑（比如复杂的加密、断言），而不是在 XML 界面里点点点，这样更利于版本控制和团队协作。”
