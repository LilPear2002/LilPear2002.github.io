MongoDB基础学习

基础概念

MongoDB 是一个文档型数据库。

  • SQL (MySQL) 像 Excel 表格，必须先定义列头（Schema），每一行格式必须一样。
  • MongoDB 像一个文件夹，里面装了很多 JSON 文件。每一个文件可以长得不一样，非常灵活。
MySQL (关系型) MongoDB (非关系型) 理解
Database (数据库) Database (数据库) 一样的，存放数据的容器。
Table (表) Collection (集合) 比如 users 集合，存放所有用户信息。
Row (行) Document (文档) 关键点！ 每一行数据是一个 JSON 对象。
Column (列) Field (字段) JSON 中的 Key。
Join (关联查询) Embedding (嵌套) 通常不怎么用 Join，而是把地址直接写在用户文档里。

在 MongoDB 中，数据是以 BSON (Binary JSON) 格式存储的，看作 JSON 就行。

这里有一个非常重要的概念：修改器 (Modifier)。

在 SQL 里可以直接 SET age=24，但在 MongoDB 里，必须使用 $set 操作符。如果不用 $set，它可能会用新数据覆盖掉整个旧文档（造成数据丢失）。

问“MongoDB 更新时 $set 是干嘛的？”

答：用于指定只更新某些字段。如果不加 $set 直接传 JSON，可能会导致整个文档被替换，丢失其他原有字段。

进阶知识理论

  1. 索引 (Index) —— 速度的来源
  2. 聚合 (Aggregation) —— 复杂的流水线
  3. 副本集 (Replica Set) —— 高可用不宕机
  4. 分片 (Sharding) —— 海量数据扩容
  5. 存储引擎 (WiredTiger) —— 底层的读写原理

索引

MongoDB 的索引原理和 MySQL（InnoDB）非常像，底层大多采用 B-Tree (B树) 结构。

  • 没有索引时：MongoDB 必须执行 Collection Scan (全表扫描)。就像你在查字典，没有拼音目录，得从第一页翻到最后一页去找“字”。
  • 有索引时：MongoDB 执行 Index Scan (索引扫描)。直接去 B-Tree 树上找到位置，瞬间拿到数据。

测开视角：

  • 复合索引 (Compound Index)：如果经常查 find({name: "A", age: 20})，那应该建立一个 (name, age) 的联合索引。
    • 注意点：最左前缀原则（和 MySQL 一样）。如你只建了 (name, age) 索引，但查询时只查 age，这个索引是失效的！
  • TTL 索引：这就很有意思了。可以给索引设置一个“过期时间”（比如 1 小时）。
    • 应用：验证码、临时 Token、日志。到了时间，MongoDB 会自动删除这些数据。这在测试“验证码过期”场景时经常用到。

聚合管道

普通的 find() 只能做简单的筛选。如果要对应 SQL 中的 Group By、Join、Sum、Avg，MongoDB 采用的是一种**“流水线” (Pipeline)** 的设计模式。

想象一个工厂流水线，数据就是原材料：

  1. $match (筛选)：先过滤掉废品（只保留 2023 年的订单）。
  2. $project (整形)：把不需要的零件拆掉（只保留“金额”和“用户ID”字段）。
  3. $group (分组)：按用户ID把订单堆在一起，算总金额。
  4. $sort (排序)：按金额从大到小排。
  5. 输出。

**测开视角：

  • 问：“MongoDB 能做多表关联查询吗？”
  • 答案是：能，但是不推荐。使用 $lookup 阶段可以实现类似 SQL Left Join 的功能，但性能开销很大。NoSQL 的设计理念本来就是“反范式化”（把数据嵌套在一起），尽量避免关联查询。

副本集

这是生产环境的标配。单机版的 MongoDB 挂了就完了，所以需要一主多从。

  • Primary (主节点)：唯一的“老大”，负责所有的写操作（增删改）。也可以读。
  • Secondary (从节点)：负责同步主节点的数据。默认不可读（为了保证强一致性），但可以配置成“只读”。
  • Heartbeat (心跳)：节点之间每隔 2 秒互通一下消息：“你还活着吗？”
  • 自动故障转移 (Failover)：如果主节点挂了，剩下的从节点会投票，选出一个新的主节点。整个过程通常在几秒内完成。

测开视角：

  • 读写分离：测试时，可以配置连接字符串，让“写”走主节点，“读”走从节点（ReadPreference）。需要测试数据同步是否有延迟（Replication Lag）。比如主节点刚写进去，立刻去从节点读，读到了吗？

分片

当数据量达到 TB 级别，或者并发量太大，一台机器（哪怕是主从架构）也扛不住了。这时候就需要分片。

Redis 也有类似的概念（Redis Cluster）。MongoDB 的分片架构包含三个角色：

  1. Mongos (路由)：这是“看门大爷”。客户端只跟它打交道。它负责把请求分发到具体的片上。
  2. Config Server (配置服)：这是“地图”。它记录了哪块数据在哪个片上。
  3. Shard (分片)：这是真正存数据的“仓库”。每个 Shard 通常也是一个副本集（保证自己不挂）。

**测开视角：

  • 分片键 (Shard Key)：这是最关键的。比如按“用户ID”分片。
    • 如果选得不好（比如按“性别”分片），会导致数据倾斜（Jumbo Chunk），所有数据都跑到其中一个片上去了，另一个片闲得慌。这是性能测试要关注的隐患。

存储引擎

MongoDB 3.2 之后默认使用 WiredTiger 引擎。只需要记住它相对于老引擎（MMAPv1）的两个巨大提升：

  1. 文档级锁 (Document-Level Concurrency)：
    • 老版本：你要修改一条数据，MongoDB 会锁住整个库或整个集合（Collection）。别人都得排队。
    • WiredTiger：只锁要修改的那一行文档。并发能力极强！
  2. 数据压缩：
    • 它在把数据存入磁盘前会进行压缩。所以在磁盘上看到的文件大小，往往比实际数据量要小很多。
