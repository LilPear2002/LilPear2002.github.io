git基础学习

在测开的实际工作中，Git 不仅仅是用来“存代码”的，还需要用它来管理自动化测试脚本、处理版本回滚、甚至配合 CI/CD（持续集成/持续部署）流水线。

基础概念

三个工作区

Git 在本地电脑上，其实把文件分成了三个状态（区域）：

  1. 工作区 (Working Directory)： 平时写代码、能看到文件的地方（比如 VS Code 打开的文件夹）。
  2. 暂存区 (Staging Area/Index)： 一个临时的“购物车”。把想提交的修改先放在这里。
  3. 版本库 (Repository/.git)： 最终的档案库，保存了所有历史版本。

问：“git add 是做什么的？为什么不直接 commit？”

答： add 允许我们要挑选一部分修改进行提交。比如改了3个文件，但只想先把其中2个相关的提交为一个版本，就可以只 add 那两个文件。这体现了提交的原子性。

  • git log --oneline：一行显示一条历史，非常简洁（推荐）。

  • 如果不通过分支： 直接在现有文件里改改改。万一写错了，或者改到一半要去修别的 Bug，代码就乱了，原来的稳定版本也跑不起来了。
  • 使用分支： 就像开启了一个**“平行宇宙”**。在这个新宇宙里，随便折腾，原来的宇宙（主分支）不受任何影响。等写好了，再把两个宇宙合并。

问：“Git 里的 HEAD 指针是什么？”

答：

“HEAD 就是一个指向当前光标的指针。当切换分支（checkout/switch）时，HEAD 就指到了那个分支的最新一次提交上。所有的 commit 操作，都是基于 HEAD 当前的位置继续向前走的。”

协作

远程仓库的核心命令（概念与流程）

在团队里，不会每个人都只在自己电脑上保存代码，而是有一个中央仓库（Remote Repository），通常托管在 GitHub 或 GitLab 上。

  1. git clone <url>
    • 场景： 入职第一天。
    • 作用： 把远程的代码完整地下载到本地电脑，生成一个文件夹。
  2. git pull
    • 场景： 每天早上开工前。
    • 作用： 把同事们昨天写的代码拉取下来，同步到本地，保持最新。
    • 口诀：Pull = Download + Merge
  3. git push
    • 场景： 写完了测试脚本，在本地 commit 过了。
    • 作用： 把你的提交推送到远程服务器，这样导师和同事才能看到你的代码。

如果push 失败，通常是因为同事刚更新了代码，而自己本地不是最新的。

原则：先 pull（同步最新），再 push（推送自己）。

解决冲突 (Merge Conflict)

冲突的本质： 两个人（或者两个分支）修改了同一个文件的同一行代码，Git 懵了，不知道该听谁的，于是把烂摊子扔给你来决定。

1. 制造冲突

  • Step 5: 修改 test_login.py 的同一行（第一行）。

    • 改为：print("User Login: V2")

    • 提交：

2. 引爆冲突

现在，feature/a 说是 V1，feature/b 说是 V2。Master 说：我要合并你们！

  • Step 6: 切回 master。

  • Step 7: 合并分支 A（这一步会顺利成功）。

  • Step 8: 合并分支 B（BOOM! 💥 冲突出现）。

    • 看到类似这样的提示：CONFLICT (content): Merge conflict in test_login.py。
    • Git ：自动合并失败，请手动修复。

3. 解决冲突

现在输入 git status，会看到 test_login.py 处于 both modified 状态。

Step 9: 打开代码编辑器，查看 test_login.py。

会看到 Git 给你留下的“记号”，长这样：

  • <<<<<<< HEAD 到 ======= 之间：是当前（Master/A）的代码。
  • ======= 到 >>>>>>> feature/b 之间：是要合并进来（B）的代码。

操作： 删除那些 <<<, ===, >>> 符号，只保留你想要的代码。

IDE中使用Git

VS Code

适用场景： Python 脚本、前端测试、轻量级项目。VS Code 的 Git 功能是开箱即用的，集成在左侧侧边栏。

  • 查看状态 (git status)：

    点击左侧的 “源代码管理” (Source Control) 图标（看起来像个树杈）。这里会列出所有 Changes（修改过的文件）。

    • M: Modified (修改)
    • U: Untracked (未追踪/新建)
    • D: Deleted (删除)

  • 暂存与提交 (git add & git commit)：

    1. 鼠标悬停在文件名上，点击 + 号 = git add (放入暂存区)。
    2. 在上方的输入框写 Commit Message。
    3. 点击输入框上方的 Commit (提交) 按钮（或者按 Cmd/Ctrl + Enter）。

  • 同步 (git pull & git push)：

    VS Code 极其贴心，提交后，左下角会显示一个 “Sync Changes” (同步更改) 按钮（或者在左侧菜单里是个刷新图标）。点击它，VS Code 会自动先 Pull 再 Push。

  • 分支切换 (git switch)：

    看窗口左下角。那里显示着当前分支名（如 main）。点击它屏幕顶部会弹出一个列表，让选择或创建新分支。

PyCharm / IDEA

适用场景： 大型自动化测试框架、Java/Spring Boot 项目。JetBrains 的 Git 功能非常强大，尤其是“对比差异”和“解决冲突”的界面。

  • 提交窗口 (add & commit)：

    默认在左侧有一个 Commit 标签页（或者按 Cmd/Ctrl + K 唤起）。

    • 它用复选框 (Checkbox) 代替了 git add。勾选文件 = 放入暂存区。
    • 在下方写 Message，然后点右下角的 Commit 按钮。
    • 技巧： 按钮旁边有个箭头，点开可以选择 Commit and Push，一步到位。

  • 拉取与推送 (pull & push)：

    右上角通常有一组 Git 按钮：

    • ↙️ (蓝色箭头向下)：Update Project (相当于 git pull)。
    • ↗️ (绿色箭头向上)：Push。

  • 分支管理：

    看窗口右下角（或者顶部导航栏右侧）。显示分支名，点击它会弹出一个强大的菜单，可以新建分支、Merge、Rebase 等。

可视化解决冲突

在命令行里看到的 <<<<<<< HEAD 那些乱七八糟的符号，在 IDE 里，再也不用看那些了。当发生冲突时，IDE 会弹出一个 “Resolve Conflicts” 窗口。

三栏视图 (3-Way Merge)：

  • 左边： 自己的代码。
  • 右边： 传入的代码 (Their Changes / From Server)。
  • 中间： 最终结果 (Result)。

只需要看着中间的结果栏。

  • 想要左边的？点击左边的 >> 箭头。
  • 想要右边的？点击右边的 << 箭头。
  • 都不想要？直接在中间那栏手动打字修改。
  • 改完点 Apply，IDE 帮你自动生成 git add 和 git commit。

git stash

CI/CD 与自动化流水线

“我不只写测试脚本，还配置了 CI 流水线，保证每次代码提交都能自动运行测试。”

CI/CD

有一个不知疲倦、24小时待命的机器人，名字叫“CI Server”（比如 Jenkins, GitLab CI, GitHub Actions）。它的工作很简单，但非常重要：

  1. 监听 (Listen)： 它死死盯着 Git 仓库。
  2. 触发 (Trigger)： 一旦发现有人 git push（提交代码）或者提了 PR。
  3. 执行 (Execute)： 它立马把代码下载下来，在它的电脑上自动安装环境，自动运行写的 pytest 脚本。
  4. 报告 (Report)：
    • 如果代码没问题：它给你打个 ✅。
    • 如果代码有 Bug（测试挂了）：它给你打个 ❌，并直接通知。

这就叫 CI (Continuous Integration - 持续集成)。 它的核心目的是：尽早发现 Bug。

它怎么“挡住”Bug

模拟一次**“拦截事故”**。

  1. 修改本地代码： 故意把测试脚本改坏。
    • 比如在 test_login.py 里写一个肯定报错的断言：assert 1 == 2。
  2. 推送代码： git push。

    • GitHub 会发邮件告诉你构建失败了。

在公司里的实际应用：通常会设置**“保护分支” (Branch Protection)。如果 CI 流水线是红色的（测试没过），GitLab/GitHub 会禁止**你把代码合并进 master 分支。必须先把测试修好。

这就是测开存在的意义：代码质量的守门员。
