Jmeter基础学习

Jmeter概述

1. JMeter 是什么？

  • 官方定义：Apache JMeter 是一个 100% 纯 Java 编写的开源软件，主要用于进行负载测试（Load Testing）\和\性能测量。
  • 测开视角的理解：它是一个接口测试 + 性能测试的容器。它通过模拟大量的并发用户（线程），向服务器发送请求，并收集服务器的响应数据，以此来评估系统的抗压能力。

2. 核心特性

A. 它不是浏览器

这是新手最容易误解的地方，也是面试最爱问的。

  • 浏览器：发送请求 -> 接收 HTML -> 解析 CSS/JS -> 渲染页面。
  • JMeter：发送请求 -> 接收 HTML -> 结束。
  • 区别：JMeter 不会执行页面上的 JavaScript，也不会渲染样式。
  • 意义：这使得 JMeter 非常轻量级，一台普通电脑就能模拟成百上千个用户。但也意味着它不能用来测 UI 界面（比如页面布局乱没乱），只能测后端接口。

B. 多协议支持

虽主要用它测 HTTP/HTTPS（Web 接口），但它其实是个“万能插头”，支持：

  • Web: HTTP, HTTPS
  • 数据库: JDBC (可以直接连数据库测 SQL 性能)
  • FTP: 文件传输
  • 消息队列: JMS, Kafka (通过插件)
  • 邮件: SMTP, POP3
  • TCP: 纯 Socket 通信

C. 可扩展性强

JMeter 提供了丰富的插件（Plugins Manager），并且支持通过 BeanShell 或 JSR223 (Groovy) 编写脚本来处理复杂的测试逻辑。如果自带功能不够用，可以写 Java 代码来扩展它。

3. 主要用它做什么？

在使用 JMeter 时，根据配置的不同（并发量、持续时间）做不同的测试：

  1. 负载测试 (Load Testing)：
    • 目的：测试系统在正常或预期的负载下能否正常工作。
    • JMeter 操作：模拟 100 个用户，持续运行 10 分钟，看响应时间是否达标。
  2. 压力测试 (Stress Testing)：
    • 目的：不断增加负载，直到系统崩溃，寻找系统的极限瓶颈。
    • JMeter 操作：从 100 用户增加到 1000、2000…直到报错率飙升，找到服务器挂掉的那个点。
  3. 稳定性/耐久性测试 (Soak Testing)：
    • 目的：长时间运行，检测是否有内存泄漏。
    • JMeter 操作：保持一定的负载，连续跑 24 小时或 7 天。

4. JMeter 的工作原理

想象一个工厂：

  • Controller (控制台)：测试人员发号施令。
  • Thread Group (线程组)：一群工人（虚拟用户）。
  • Sampler (取样器)：工人做的具体动作（比如“发送登录请求”）。
  • Timer (定时器)：工人的动作频率（比如“思考 1 秒再点”）。
  • Assertion (断言)：质检员（检查返回结果对不对）。
  • Listener (监听器)：报表员（生成图表和报告）。

基础环境与核心元件

第一步：启动与创建测试计划

  1. 进入 JMeter 解压目录的 bin 文件夹。
    • Windows: 双击 jmeter.bat。
    • Mac/Linux: 在终端运行 sh jmeter.sh。
  2. 看到图形界面后，会看到左侧默认有一个 Test Plan (测试计划)。
    • 这是所有测试元素的容器，相当于一个项目文件。

第二步：创建虚拟用户池 (Thread Group)

需要先定义“谁”在测。

  1. 右键点击 Test Plan -> Add (添加) -> Threads (Users) (线程) -> Thread Group (线程组)。 JMeter 是由 Java 实现的，并且使用一个 Java 线程来模拟一个用户，因此线程组（Thread Group）就是指一组用户的意思
  2. 右侧会出现配置面板，目前保持默认即可：
    • Number of Threads: 1 (虚拟用户数)
    • Ramp-Up Period: 1 (1秒内启动完)
    • Loop Count: 1 (只跑一次)

第三步：模拟发送请求 (Sampler)

  1. 右键点击 刚才创建的 Thread Group -> Add -> Sampler (取样器) -> HTTP Request (HTTP请求)。
  2. 在右侧面板配置以下信息：
    • Protocol [http]: 输入 https
    • Server Name or IP: 输入 www.baidu.com (注意：不要带 https:// 前缀，也不要带后面的 /)
    • Port Number: 留空 (默认443)
    • Method: 选择 GET
    • Path: 输入 / (代表访问根路径)

第四步：安装监控探头 (Listener)

请求发出去后，我们需要看到结果。

  1. 右键点击 Thread Group -> Add -> Listener (监听器) -> View Results Tree (察看结果树)。
    • 解释：这是调试脚本最常用的组件，能看到详细的请求和响应数据。

第五步：运行与验证

  1. 点击工具栏顶部的 绿色三角形 (Start) 按钮。
  2. 系统会提示保存脚本，点击 Yes，随便存个位置（例如 baidu_test.jmx）。
  3. 点击左侧的 View Results Tree。
  4. 看右侧的结果：
    • 绿色盾牌 ✅：表示请求成功（HTTP 状态码 200）。
    • 红色盾牌 ❌：表示失败。

Sample Start (开始时间) JMeter 发出请求的那个毫秒级时间戳。

Latency (延迟 / TTFB)：它不是单纯的“延迟”,是 Time To First Byte (TTFB)。

  • 定义：从发送请求开始，到接收到服务器返回的第一个字节的时间。
  • 意义：这代表了网络传输时间 + 服务器处理时间（查库、运算）。

Load Time (加载时间 / Elapsed Time)：

  • 定义：从发送请求开始，到接收完最后一个字节的时间。
  • 公式：Load Time = Latency + 数据下载时间。

参数化与模拟搜索

任务一：修改脚本进行搜索

百度搜索的原理其实是访问 /s 这个路径，并带上 wd=关键字 这个参数。

  1. 回到 HTTP Request。
  2. 修改配置：
    • Path: 修改为 /s (原本是 /)。
  3. Parameters (参数) 部分（在 Path 下面）：
    • 点击 Add (添加)。
    • Name: 输入 wd (百度的搜索参数名)。
    • Value: 输入 JMeter性能测试 (或者你想搜的任何词)。
  4. 运行脚本，查看 View Results Tree。
    • 点击请求，看 Request Body 或 URL，应该能看到 https://www.baidu.com/s?wd=JMeter性能测试。

任务二：参数化 (CSV Data Set Config)

如果要测试 100 个词，总不能复制 100 个 HTTP Request 吧？要把数据和脚本分离。

1. 准备数据文件

  • 在电脑上新建一个 search_words.txt。

  • 里面写三行内容（不要表头）：

2. 配置 CSV 数据读取器

  • 右键点击 Thread Group -> Add -> Config Element (配置元件) -> CSV Data Set Config。
  • 配置面板：
    • Filename: 点击 Browse 选择刚才的 search_words.txt。
    • Variable Names (comma-delimited): 输入 keyword (这是给每一行数据起的变量名)。
    • Ignore first line: False (因为没有写表头)。

3. 引用变量

  • 回到 HTTP Request。
  • 找到刚才填写的 wd 参数。
  • 把 Value 里的 JMeter性能测试 删掉，改成 ${keyword}。
    • 解释：在 JMeter 里，${变量名} 是引用变量的标准语法。

4. 调整并发数

  • 点击 Thread Group。
  • 把 Number of Threads 改成 3 (因为我们有 3 个词)。
  • 把 Loop Count 保持 1。

5. 运行与验证

  • 点击运行。
  • 查看 View Results Tree。
  • 你应该看到 3 个 HTTP Request。
  • 分别点开它们的 Request 标签，检查 URL。
    • 第一个应该是 wd=python
    • 第二个应该是 wd=java
    • 第三个应该是 wd=selenium

问：如果将 Number of Threads 设为 1，但把 Loop Count 设为 3，效果和上面一样吗？

答：如果把线程数设为 1，循环次数设为 3，它依然会依次读取 python, java, selenium。因为 CSV Data Set Config 是一个“全局发牌员”。不管是哪个线程、第几次循环来找它要数据，它都会把指针往下移一行。

但它们在“性能测试”中的意义完全不同：

  • 3 线程，1 循环：是 3 个人同时 访问百度。这叫并发（Concurrency），用来测试服务器抗压能力。
  • 1 线程，3 循环：是 1 个人连续 访问 3 次。这叫串行（Serial），压力很小，通常用于功能测试或稳定性测试。

关联

在真实业务中，接口之间是有联系的。

  • 场景：用户登录 -> 服务器返回一个 Token -> 用户拿着 Token 去查余额。
  • 问题：Token 是动态生成的，没法写死在脚本里。
  • 解决：需要从第 1 个接口的响应中提取出 Token，存成变量，传给第 2 个接口。

用一个专门的测试靶场：httpbin.org。

第一步：准备环境

  1. 把之前的百度请求禁用（右键 -> Disable）。
  2. 创建一个新的 Thread Group。

第二步：创建接口 A (获取数据)

  1. 添加 HTTP Request。
  2. 配置：
    • Server Name: httpbin.org
    • Method: GET
    • Path: /uuid
  3. （建议先运行一下，在察看结果树里看 Response Body，会看到类似 {"uuid": "53e4..."} 的 JSON 数据）。

第三步：添加提取器 (JSON Extractor) —— 关键步骤

要把uuid抓出来。

  1. 右键点击 刚才的“接口 A” -> Add -> Post Processors (后置处理器) -> JSON Extractor。
  2. 配置面板：
    • Names of created variables: my_uuid (给提取出来的东西起个变量名)。
    • JSON Path expressions: $.uuid
      • 解释：$ 代表根节点，.uuid 代表找 key 为 uuid 的值。这是标准的 JSONPath 语法。
    • Match No.: 1 (取第 1 个匹配到的)。

第四步：创建接口 B (使用数据)

  1. 在线程组下再添加一个 HTTP Request。
  2. 配置：
    • Server Name: httpbin.org
    • Method: POST (注意是 POST)
    • Path: /post
  3. Body Data (点击 Parameters 旁边的标签)：
    • 输入：{ "id": "${my_uuid}" }
    • 解释：我们要把刚才提取出来的变量，拼装成 JSON 发回去。

第五步：验证

  1. 运行脚本。
  2. 查看 View Results Tree。
  3. 点开 接口 B 的请求。
    • 点击 Request Body：看 ${my_uuid} 是否变成了具体的字符串（如 c8f1...）？
    • 点击 Response Body：httpbin 会把发的数据原样返回，检查里面是否有刚才发的 ID。

调试和断言

Debug Sampler

在开发脚本时，经常不知道变量到底提取到了没有。靠猜是不行的，需要看内存里的变量。

  1. 添加调试取样器：
    • 右键点击 Thread Group -> Add -> Sampler -> Debug Sampler。
    • 把这个 Sampler 拖到所有请求的最后面。
  2. 运行脚本：
    • 点击运行，查看 View Results Tree。
    • 点击那个绿色的 Debug Sampler，查看 Response Body。
    • 你应该能看到类似 my_uuid= 或者 my_uuid=ERROR (如果设置了默认值) 或者根本找不到 my_uuid。

诊断：如果 my_uuid 后面是空的，说明 JSON Extractor 没有提取到数据。原因通常是：

  • JSON Path 写错了：比如写成了 $.id 但实际返回是 uuid。
  • Default Value：你在提取器里把默认值留空了。
    • 建议习惯：在 JSON Extractor 的 Default Value 里填上 NOT_FOUND。如果提取失败，变量就会变成 “NOT_FOUND”，在 Debug 时一眼就能看到。

修正提取器

  1. 回到 JSON Extractor。
  2. 检查 JSON Path expressions：必须是 $.uuid (大小写敏感)。
  3. 设置 Default Value：输入 NOT_FOUND。
  4. 再次运行，看 Debug Sampler 里的 my_uuid 是否变成了具体的字符（如 53e4...）。

Response Assertion

让脚本在数据错误时自动报错（变红），而不是人工去检查。给 接口 A 加一个“安检员”，确保它返回了想要的数据。

  1. 右键点击 接口 A (获取 UUID 的请求) -> Add -> Assertions (断言) -> Response Assertion (响应断言)。
  2. 配置面板：
    • Apply to: Main sample only (默认)。
    • Field to Test: 选择 Text Response (检查响应文本)。
    • Pattern Matching Rules: 选择 Contains (包含)。
    • Patterns to Test: 点击 Add，输入 uuid。
    • 解释：这意味着，如果返回结果里连 “uuid” 这个词都没有，就判定为失败。

更高级的用法（检查接口 B）： 我们要确保接口 B 发送的数据不是空的。

  1. 右键点击 接口 B -> Add -> Assertions -> Response Assertion。
  2. Field to Test: 改选 Request Headers 或者 Request Data (注意这里是检查请求，不是响应)。
    • JMeter 的断言通常检查响应，但也可以检查请求。不过更常用的做法是在提取后立刻检查变量。
    • 更好的做法：在接口 A 下面直接加一个 JSR223 Assertion (代码断言)，判断 ${my_uuid} 是否为空。不过为了不增加难度先跳过代码部分。

场景设计与“无界面”压测

问：“怎么执行 1000 并发的压测的？” 如果回答：“点击了界面上的绿色启动按钮” 这是不对的

原因：JMeter 的图形界面 (GUI) 非常消耗内存和 CPU。真正的高并发压测，必须使用命令行 (CLI / Non-GUI) 模式。

任务一：事务控制器 (Transaction Controller)

假设一个“购买”操作包含了 3 个 API 请求。老板只想知道“购买”这个动作慢不慢，不想看每个 API 的细节。需要把它们包起来。

  1. 右键点击 Thread Group -> Add -> Logic Controller (逻辑控制器) -> Transaction Controller。
  2. 配置面板：
    • Generate parent sample: 勾选它。
      • 作用：勾选后，报告里只会显示“Transaction Controller”这一行数据，把内部细节隐藏起来（或者作为子节点），方便看总耗时。
  3. 移动请求：
    • 把之前的 接口 A (GET uuid) 和 接口 B (POST) 拖动到这个 Transaction Controller 的里面（成为子节点）。
    • 现在的结构应该是：Thread Group -> Transaction Controller -> [接口A, 接口B]。

任务二：准备压测参数

为了模拟一点压力，调整一下线程组参数：

  1. 点击 Thread Group。
  2. Number of Threads: 10 (模拟 10 个用户)。
  3. Ramp-up Period: 2 (2秒内启动完)。
  4. Loop Count: 5 (每人跑 5 次)。
  5. 保存脚本：一定要保存！假设保存为 test_stage3.jmx，记住保存的路径

任务三：命令行执行 (CLI Mode) —— 核心技能

  1. 进入脚本所在目录并打开终端/命令行

  2. 输入压测命令：

    参数详解：

    • -n: non-GUI mode (无界面模式)。
    • -t: 指定测试脚本文件 (test plan)。
    • -l: 指定结果文件路径 (log)，通常存为 .jtl 格式。
    • -e: 测试结束后生成 Web 报告 (end of run)。
    • -o: 指定报告输出文件夹 (output folder)。注意：这个文件夹必须是空的或者不存在，否则会报错。

  3. 执行：

    • 按下回车。应该会看到命令行里在滚动类似 Summary + ... 的日志。等待它显示 ... end of run。

任务四：查看上帝视角报告

  1. 打开脚本目录。
  2. 多了一个 report 文件夹。
  3. 进入文件夹，双击 index.html。

这是一个非常专业的 HTML 性能测试报告。找到 Charts -> Response Times -> Response Time Percentiles。

解读Statistics 表格核心指标

  1. Samples (样本数): 总请求数。
  2. Average (平均值): 平均分。
    • 误区：平均值是最会骗人的！ 比如我和马云的平均资产是几千亿，但这不能代表我的真实水平。在性能测试中，只要有几个极端的慢请求（比如超时），平均值就会被拉高或拉低，掩盖真相。
  3. Min / Max: 响应最快和最慢的时间
  4. Error %: 报错的请求占比。通常要求 0% 或者 < 0.1%。
  5. Throughput (吞吐量/TPS): 代表服务器“每秒能处理多少个请求”。

Percentiles (百分位数 - P90, P95, P99)

互联网大厂定验收标准（SLA） 的依据。

  • P90 (90th pct)：将所有请求的响应时间从小到大排列，排在 90% 位置的那个时间。
    • 人话：“90% 的用户都能在 X 秒内得到响应。”
  • P99 (99th pct)：排在 99% 位置的时间。
    • 人话：“99% 的用户都能在 X 秒内得到响应。” (排除了 1% 的极端慢请求)。

如果 Average 是 100ms，但 P90 是 500ms，说明了什么？

结论：系统不稳定，存在严重的“长尾效应” (Long Tail)。

  • 分析：
    • 平均值 100ms 很低，说明大部分请求都非常快（可能只有 50-80ms）。
    • 但 P90 到了 500ms，说明有至少 10% 的用户体验非常差，比正常慢了 5 倍以上。
  • 常见原因：
    • 代码问题：偶尔触发了复杂的逻辑分支。
    • Java 应用：发生了 Full GC (垃圾回收)，导致系统暂停了几百毫秒。
    • 数据库：偶尔遇到了锁等待，或者缓存刚好失效了（缓存穿透）。

“不能只看平均值，因为平均值会掩盖性能抖动。在这个场景下，虽然平均响应快，但 P90 差距过大，说明系统存在偶发性的卡顿或资源争用，不满足稳定性要求，需要排查 GC 日志或慢查询。”

监控与瓶颈定位

**如果分数低（响应慢），到底是谁的锅？**是网络卡？CPU 满了？还是数据库死锁了？

JMeter 本身只能告诉你“慢”，不能告诉你“为什么慢”。需要给服务器插上“体温计”。

任务一：安装插件管理器 (Plugins Manager)

JMeter 原生功能有限，需要安装最强的插件来监控服务器。

下载插件管理器：去官网下载 jmeter-plugins-manager-1.x.jar

**替代方案： 不依赖 JMeter 插件，而是学习用 **Linux 命令来监控。

任务二：模拟瓶颈与观察 (思想实验)

发现 TPS 上不去了，P99 特别高。需要立即去服务器上敲以下命令：

  1. top —— 看 CPU 和 内存
    • 如果你看到 us (User CPU) 飙到 90% 以上：说明是代码计算量太大（算法不行，或者死循环）。
    • 如果你看到 wa (Wait I/O) 很高：说明 CPU 都在等磁盘读写（数据库在疯狂刷盘，或者日志打得太疯）。
  2. free -m —— 看内存
    • 如果 available 几乎为 0，且 Swap 被大量使用：说明内存泄漏 (Memory Leak) 或内存不足。
  3. netstat -nat | grep ESTABLISHED | wc -l —— 看网络连接
    • 查看当前建立了多少连接。如果连接数过多但处理不过来，可能是连接池配置太小。

问：如果压测时，JMeter 这一端的 CPU 占用率达到了 100%，而服务器端 CPU 只有 10%。这时候生成的测试报告可信吗？

答：绝对不可信，报告是废的。

  • 原因：这叫“施压机瓶颈”。
    • 就像想测试法拉利（服务器）能跑多快，但骑着自行车（JMeter）去追。
    • 因为你的自行车最快只能骑 20km/h（CPU 满了，发不出更多请求了），所以会误以为法拉利最高时速只有 20km/h。
    • 实际上服务器可能根本没感觉。
  • 解决办法：
    1. 优化脚本（关掉 View Results Tree，用命令行跑）。
    2. 分布式压测：找 3 台电脑一起向服务器发起攻击
