力扣SQL题目总结

lc181

相关子查询

问题：这是相关子查询的写法，会对e 表中的每一行，都去执行一次子查询。如果有 N 个员工，这个子查询就要被触发 N 次。如果 N 是 10 万，就要查询 10 万次。这就是经典的 N+1 问题，性能较低。

外连接

左连接会保留所有员工，如果managerId是NULL，e2.salary 也会是 NULL，那么e1.salary > NULL 依然是 UNKNOWN，而WHERE 子句过滤了所有结果为 UNKNOWN 的行。

内连接

  • SELECT … FROM A, B WHERE A.id = B.id 是 隐式内连接。
  • SELECT … FROM A JOIN B ON A.id = B.id 是 显式内连接。

虽然它没有写 JOIN 关键字，但它在逻辑上满足了内连接的定义：只保留满足 WHERE 子句中连接条件 不推荐这种写法

注意，JOIN 等同于 INNER JOIN：在 SQL 中，JOIN 关键字默认就是内连接 只有当 a.ManagerId = b.Id 这个条件成立时，才会保留这一行。 在本题中，内连接和外连接都能得到正确结果

lc 182、1075、1050

一旦使用了 GROUP BY，SELECT 后面只能跟两类东西（标准SQL）：

  1. 参加分组的字段（即 GROUP BY 后面的那个字段，如 module）。
  2. 聚合函数（被捏扁后的统计结果，如 COUNT(), MAX(), AVG()）。

此外，GROUP BY 后面支持跟 1 个、2 个或者多个字段，用逗号分隔就行

lc 183

左连接 普遍写法

同样的问题，需要执行子查询获取所有 customerId，子查询的时间复杂度较高

理解 EXISTS 的核心在于：它不关心子查询返回了什么数据，它只关心有没有数据。

如果子查询结果集大，EXISTS 在找到第一个匹配后立即返回，在某些数据库中对大表可能比 LEFT JOIN 更好且对 Orders.customerId 有索引时效率高。

lc 197

类型 示例 是否支持
DATE ‘2026-03-25’ ✅ 完全支持
DATETIME ‘2026-03-25 14:30:00’ ✅ 支持，只比较日期部分
TIMESTAMP ‘2026-03-25 14:30:00’ ✅ 支持，只比较日期部分
字符串（日期格式） ‘2026-03-25’ ✅ 会自动转换
