linux基础学习

Linux介绍

简单说，Linux 和平时用的 Windows、macOS 一样，都是操作系统（Operating System）。

但它有两个核心特点，注定让它成为了服务器领域的霸主：

  1. 开源免费：源代码是公开的，任何人都可以修改。
  2. 极度稳定：它可以连续运行几年不关机、不重启，而且效率极高。

在 Windows/Mac 电脑上写代码、写文档（这是工作环境）；但测试的代码、数据库、服务，通常都运行在远端的 Linux 服务器上（这是生产/测试环境）。

可以把 Linux 系统想象成一家餐厅：

  • 硬件（Hardware） ➜ 厨房的食材和灶具
    • 这是底层的 CPU、内存、硬盘。
  • 内核（Kernel） ➜ 大厨
    • 这是 Linux 的核心。只有大厨（内核）有权直接操作灶具（硬件），别人不能乱动。它负责管理内存、CPU 调度。
  • Shell（终端/命令行） ➜ 服务员
    • 这就是以后要面对的那个“黑底白字的窗口”。
    • 顾客不能直接进厨房炒菜，必须把需求告诉服务员（输入命令），服务员传达给大厨（内核），大厨做好后，服务员再显示结果。
  • 应用程序（Applications） ➜ 菜品
    • 在上面跑的 MySQL 数据库、Tomcat、Nginx 或者写的 Python 自动化脚本。

Linux 发行版（Distributions）

Ubuntu、CentOS、RedHat、Debian… 其实，Linux 内核 只有一个。但是不同的公司为了方便用户使用，把内核 + 各种常用软件 + 安装工具打包在一起，就变成了发行版。

  • CentOS / RedHat：企业级服务器**最常用的版本。特点是稳，但软件版本比较老。
  • Ubuntu：开发者和 AI 领域最喜欢的。界面友好，软件比较新。
  • Kali Linux：黑客和安全测试人员专用的（如果你做安全测试会用到）。

命令是通用的。学会了 CentOS，Ubuntu 也能上手，反之亦然。

为什么测开不用图形界面（GUI）

在 Linux 服务器上，通常通过 SSH（远程连接） 对话。

  1. 省资源：图形界面非常吃内存和 CPU。服务器资源要留给跑代码，而不是用来渲染漂亮的窗口。
  2. 效率：
    • 想删除 1000 个特定的日志文件。
    • Windows：按住 Ctrl 选半天，或者框选。
    • Linux：一行代码 rm *.log，0.1 秒搞定。
  3. 自动化：测开的核心是自动化。图形界面的操作很难写进脚本里，但命令行（Shell 脚本）天生就是为了自动化而生的。

问题：为什么说 Linux 没 GUI，但有时候在双系统/租云服务器时又能看到界面？

Linux内核 本身只是一个命令行（文本界面）的核心。它的主要职责是管理硬件和资源（进程、内存、设备等），不负责绘制窗口、按钮、菜单或处理鼠标点击。而GUI是它们在Linux内核之上，叠加了一整套庞大的“图形化软件栈”。这套软件栈不是内核的一部分，而是运行在内核之上的应用程序和服务。简单来说，图形界面只是一个普通的软件。

云服务器

简单理解为：一台位于远程数据中心、24小时不断电、通过网络租给你用的高性能电脑。

在互联网公司（比如你要去的实习岗位），所有的代码、数据库、测试环境，都不会跑在开发人员自己的笔记本上，而是跑在云服务器上。

  • 随时随地访问：合上笔记本回家了，服务器还在运行自动化测试脚本。
  • 公网 IP：云服务器有一个公网 IP 地址。在上面部署一个网站，全世界的人都能通过浏览器访问它。
  • 弹性：今天测试压力小，租个 2核4G 的；明天搞双十一压测，一键升级成 64核128G 的。

2. 打个比方

  • 虚拟机/WSL（本地环境）：就像在自己家里买了一台跑步机。如果不维护（关机/重装系统），它就停了。
  • 云服务器：就像办了一张健身房年卡。场地、设备人家都维护好了，只需要去“用”就行，而且那是公共场所（公网），可以约朋友（用户）一起来。

基础命令

目录相关命令

导航

Linux 是一个倒挂的大树，树根是 /。

  • pwd (Print Working Directory)：显示当前路径。
    • 输入 pwd 回车。应该在 /root
  • ls (List)：看看这里有什么。
    • 试一试：输入 ls，或者更常用的 ls -l (显示详细信息)。
  • cd (Change Directory)：传送门。
    • 试一试：
      1. cd / (回到根目录，这里是系统的起点)
      2. cd ~ (波浪号代表回家。无你在哪里，一键回到家目录)

文件操作

创建文件

  • mkdir (Make Directory)：建文件夹。
    • 试一试：mkdir my_test_project
  • touch：建空文件（或者更新文件时间戳）。
    • 试一试：cd my_test_project (进入刚才建的目录)，然后 touch readme.txt
  • cp (Copy)：复制（备份神器）。
    • 试一试：cp readme.txt readme.txt.bak
    • 场景：修改生产环境配置前，必须先 cp 备份一份，防止改挂了回不去。
  • mv (Move)：移动，或者重命名。
    • 试一试：mv readme.txt config.txt (把 readme.txt 改名为 config.txt)
    • mv和cp命令在移动或复制过程中都可以重命名
  • rm (Remove)：删除（慎用！）。
    • 试一试：rm readme.txt.bak (删除刚才那个备份文件)
    • 高危警告：rm -rf 目录名 是强制删除目录及其下所有东西。千万不要手滑敲成 rm -rf /，那样服务器就“自杀”了。

文件编辑

在 Linux 上用 Vim。 Vim 有两个模式

  1. 命令模式（默认）：键盘是用来发号施令的（复制、粘贴、保存），不能打字。
  2. 编辑模式：这才是打字的。

    • 解释：:是指令开始，w=write(保存)，q=quit(退出)。

如果不小心搞乱了想强制退出（不保存）：按 Esc，输入 :q! 然后回车。

tar 是 Linux 最经典的打包命令，通常打包出来的文件叫 .tar.gz。只需要记住两句口诀：

  • 打包（压缩） ➜ czvf

    • 记忆法：Create (创建) Zipped (压缩) Verbose (显示过程) File (档案)。
    • 意思：把当前目录下的 logs 文件夹，打包成 my_logs.tar.gz。
  • 解包（解压） ➜ xzvf
    • 命令：tar -xzvf my_logs.tar.gz
    • 记忆法：Xtract (提取/解压)。把 C 换成 X 就行了。

权限管理

Linux 哪怕只有一个人用，它也会假想出三种角色。执行 ls -l，会看到类似 -rwxr-xr-- 的代码，它是这样分组的：

  1. Owner (属主/u)：文件的亲爹。通常是创建这个文件的人。拥有最高话语权。
  2. Group (属组/g)：亲友团。属于这个组的人，共享同一套权限。（比如“开发组”的几个人都能改代码，但不能删）。
  3. Others (其他人/o)：路人甲。除了上面两种人之外的所有人。

RWX

这是最基础的，但要注意：对文件和对目录，它们的含义略有不同！

权限符 对应数字 对文件的含义 对目录的含义 (⚠️重点)
r (Read) 4 可以查看内容 (cat) 可以列出里面有什么文件 (ls)
w (Write) 2 可以修改内容 (vim) 可以在里面创建或删除文件 (touch/rm)
x (eXecute) 1 可以运行它 (如果是脚本) 可以进入该目录 (cd)
- 0 无权限 无权限

如果一个目录有 r (读) 权限，但没有 x (执行) 权限，能 cd 进去吗？

答案：不能。你能看到里面有啥，但你进不去。

chmod (改权限)

数字法

这是基于二进制计算的：R=4, W=2, X=1。

想要什么权限，就把数字加起来。

  • 7 = 4+2+1 (读+写+执行，全能)
  • 6 = 4+2 (读+写)
  • 5 = 4+1 (读+执行)
  • 0 = 什么都不行

常见组合：

  • chmod 777 file：每个人都能读写执行（极度不安全，除非测试需要）。
  • chmod 755 file：
    • Owner (7): 啥都能干。
    • Group (5): 只能看和运行，不能改。
    • Others (5): 只能看和运行，不能改。
    • (这是脚本/程序的标准权限)
  • chmod 644 file：
    • Owner (6): 读写。
    • Group (4): 只读。
    • Others (4): 只读。
    • (这是普通配置文件的标准权限)
字母法（适合微调）

公式：u/g/o (谁) +/- (加减) r/w/x (权限)

  • chmod u+x run.sh：给Owner增加执行权限。
  • chmod go-w config.ini：把Group和Others的写权限去掉（防止别人乱改）。

终极权限：Sudo & Root

  • Root：Linux 的神。无视所有规则。即使文件是只读的，Root 也能强行删除。
  • Sudo：当用普通用户（比如 ubuntu）登录时，想干只有 Root 才能干的事（比如安装软件 apt install），你就要加 sudo。
    • 它的意思是：“以超级管理员的身份执行这条命令”。

进程与端口管理

  • “我部署了服务，但网页打不开，是没启动吗？”
  • “我想启动一个新的测试环境，系统提示端口被占用（Address already in use）。”
  • “服务器 CPU 突然飙到 100%，是谁在搞鬼？”

解决这些问题，需要掌握这套“查进程 -> 杀进程 -> 查端口”的连招。

查看进程

在 Windows 任务管理器里可以看到运行的所有软件。在 Linux 里，用 ps (Process Status)。

核心命令：ps -ef

这会列出系统当前运行的所有进程。但因为进程太多，直接敲这个命令会刷屏。所以配合 grep：

查看 python 程序有没有在运行。

会看到类似这样的很多行：

root 1234 1 0 10:00 ? 00:00:00 python3 server.py

  • root: 谁启动的
  • 1234 (PID): 身份证号 (Process ID)。这是操作它的唯一凭证，后面傻进程全靠它。
  • python3 server.py: 启动命令是什么？

💡 注意：通常会看到两行，其中一行是 grep python 本身，那是你刚才敲的命令，忽略它。

进阶命令：top

如果想看“谁最吃 CPU”或者“内存还剩多少”，输入 top。

  • 这就相当于 Windows 任务管理器的“性能”页。
  • 按 q 退出。

结束进程

当某个服务卡死了，或者想重启它，需要先把它“杀掉”。

核心命令：kill

  • 温柔一刀：kill PID
    • 例如：kill 1234
    • 这是发送信号让程序“收拾收拾自己退出”，大部分正常程序会听话。
  • 强制处决：kill -9 PID
    • 例如：kill -9 1234
    • -9 代表“强制杀死”。不管程序正在干嘛，直接从内存里抹去。当程序死锁无响应时，用这个。

查看端口

有时候进程在运行，但网页打不开，可能是**端口（Port）**没对上，或者被防火墙挡了。

核心命令：netstat 或 ss

注：如果提示 netstat: command not found，请先运行 apt install net-tools 安装它。

想知道 8080 端口现在是被谁占用的

*(记忆口诀：nlp = Numeric(数字显示) Listening(监听中) Program(显示程序名))*

输出解读：tcp 0 0 0.0.0.0:8080 0.0.0.0:* LISTEN 5678/java

  • 说明 8080 端口正在被 PID 为 5678 的 java 程序占用。
  • 如果想启动新程序也用 8080，就得先 kill 5678。

监控与分析

实时监控 (tail -f)

场景：在前端点一个按钮（比如“提交订单”），页面转圈圈没反应。需要立刻知道后台此刻报没报错。

这时候用 cat 是不行的，因为文件是静态的。需要动态地盯着它。

  • 命令：tail -f 文件名
    • -f = follow（跟随）。
    • 效果：屏幕会卡住，一旦日志文件有新内容写入，屏幕上会立刻滚动显示出来。就像你看直播弹幕一样。
  • 退出：按 Ctrl + C。

多维过滤 (Advanced grep)

第一阶段学了基础的 grep，但实战中情况更复杂。

  • 场景 A：不仅想看 Error，还想看 Warning
    • 命令：grep -E "Error|Warning" server.log
    • 解释：-E 开启扩展正则，| 代表“或者”。
  • 场景 B：报错太多了，想统计一下到底报了多少次
    • 命令：grep "Error" server.log | wc -l
    • 解释：| 是管道符（把前一个命令的结果传给后一个命令）；wc -l (Word Count -lines) 是之前学的数行数。
    • 结果：直接输出一个数字，比如 520。

Screen vs Nohup

screen 和 nohup 是两个流派，它们的目的都是一样的：让程序在你断开 SSH 连接后继续跑

Nohup —— “监控录像模式”

  • 原理：把程序往后台一丢，告诉它：“把日志写到 app.log 也就是那个文件里，别烦我。”
  • 优点：命令简单，不需要安装额外软件，不仅适合人敲，也适合写在自动化脚本里（Jenkins 只能用这个）。
  • 缺点：一旦后台运行了，就“看不见”它的实时控制台了。想看发生了什么，必须用 tail -f app.log 去查日志文件（就像去查监控录像）。
  • 场景：正式的生产环境部署、自动化部署。

Screen —— “电视直播模式”

  • 原理：screen 会在 Linux 里开一个虚拟窗口（你可以理解为开了一个分身）。
    • 你在这个分身里运行 java -jar，能直观地看到控制台在刷屏。
    • 当按快捷键（Ctrl+A+D）离开时，这个分身窗口并没有关，只是被最小化了。
    • 下次回来，输入 screen -r，那个窗口又弹出来了，一切还在继续，仿佛从未离开。
  • 优点：所见即所得。可以随时切回去看现场直播，甚至可以在运行途中暂停程序输入指令。
  • 缺点：需要额外安装 (apt install screen)；如果忘了关掉无用的 screen 窗口，服务器内存会被吃光；不适合自动化脚本。
  • 场景：调试 Bug、跑那种耗时几个小时的一次性脚本（比如导数据）

云监控

1. 什么是云监控

核心功能：它是阿里云自带的监控工具，用来实时盯着服务器（ECS）、数据库（RDS）等资源的健康状况 。

  • 作用：就像医院的心电监护仪。如果不装这个，服务器死机了你都不知道，只能等用户投诉。

  • 关键动作：

    安装插件 (Agent)：有些数据（比如 CPU）阿里云能直接看到，但有些数据（比如内存使用率）需要你在服务器里装一个小插件（Agent）才能采集到 。

    可视化大屏：可以自定义一个仪表盘，把 CPU、内存、网络流量画成折线图，看起来更直观 。

2. 告警服务

这是做稳定性测试时必须要用的功能。

  • 设置规则：可以定义“什么情况算异常”。

    例子：连续 1 分钟 CPU > 80% -> 警告；连续 5 分钟 > 95% -> 紧急报警 。

    通知方式：一旦触发规则，系统会自动发短信、邮件，或者通过 WebHook（比如钉钉群机器人）通知你 。

    一键报警：对于新手，阿里云提供了一键开启常见规则的功能（比如 CPU、磁盘、带宽满了自动报警） 。

3. 系统运维常见问题排查

A. CPU 负载过高 (CPU Load)

  • 如何判断：

    • 使用 top 或 htop 命令 。

      关键指标：Load Average（平均负载）。如果这个数值超过了 CPU 核心数的 0.5 倍，就要警惕；超过 1 倍 就是负载高了 。

    常见原因 & 解决 ：

    1. 代码死循环/计算密集：某个 Java 进程 CPU 飙升。（解决：top 找到 PID -> kill -9）。
    2. I/O 瓶颈：磁盘读写太慢，导致 CPU 在等待。（解决：换 SSD 或优化数据库查询）。
    3. 僵尸进程：进程死了但资源没释放。（解决：重启或清理依赖）。
    4. 被攻击：CPU 高但找不到进程，可能是中了挖矿病毒。

B. 内存爆满 (Memory Full)

  • 如何判断：

    • 使用 top 查看。如果 free（空闲内存）几乎为 0，或者使用率持续接近 100% 。

    常见原因 & 解决 ：

    1. 内存泄漏 (Memory Leak)：这是测开最需要抓的 Bug！程序只申请内存不释放。
    2. 配置不当：Java 的堆内存（Heap）设置得比服务器物理内存还大。
    3. 僵尸文件：文件被删了，但进程还占用着句柄，导致空间不释放。

4. 巡检报告

  • 这是一个体检报告，包含性能监控、安全风险（有没有被 DDoS 攻击）、资源配置分析（磁盘是不是快满了）等 。

测开视角：

  1. 性能测试 (Performance Testing)：
    • 当用 JMeter 对接口发起高并发压测时，必须盯着这些监控图表。
    • 场景：如果 TPS（每秒处理事务数）上不去，是因为 CPU 满了？还是因为内存漏了？或者是带宽不够了？这就叫“性能瓶颈分析”。
  2. Bug 定位：
    • 开发说：“我本地跑没问题啊。”
    • 甩出一张监控截图：“每次调这个接口，内存就涨 500M 且不回落，明显的内存泄漏。”（这比只会说“程序崩了”要专业得多）。
  3. 监控脚本：
    • 甚至可以写自动化脚本，利用阿里云的 API 去查询这些监控数据，如果发现测试环境资源不足，自动停止测试，防止误报。
